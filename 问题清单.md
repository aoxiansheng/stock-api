## 详细说明

### Critical

#### C1 `unsubscribeStream()` 先删状态后执行外部副作用，失败无回滚

- 位置: `src/core/01-entry/stream-receiver/services/stream-receiver.service.ts:617-723`
- 证据:
  - `617` 行先执行 `this.clientStateManager.removeClientSubscription(clientId, standardSymbols);`
  - 之后才执行 `scheduleRelease()`、`unsubscribeFromSymbols()`、`leaveClientFromRooms()`
  - 任一后置动作失败后进入 `717-723` 的 `catch`，仅日志并重新抛错，没有恢复客户端订阅状态，也没有取消已安排的 release
- 风险:
  - 本地视图已退订，但上游可能仍持有订阅
  - 已安排的 grace release 可能继续触发，进一步扩大状态漂移
  - 后续 `getClientCountForUpstream()`、房间路由和重连判断都会基于错误状态工作
- 归类理由:
  - 这是明确的状态一致性缺陷，且影响核心订阅生命周期，因此归为 `Critical`

### Important

#### I1 `setupDataReceiving()` 缺少幂等保护

- 位置: `src/core/01-entry/stream-receiver/services/stream-ingress-binding.service.ts:29-58`
- 证据:
  - 每次调用都直接执行 `connection.onData/onError/onStatusChange`
  - 当前 `subscribeStream()` 与 `executeClientReconnect()` 都会调用 `setupDataReceiving()`
  - 代码中没有任何 `WeakSet/Map` 或连接级标记，无法阻止对同一连接重复绑定
- 风险:
  - 同一条流数据被重复推入 `batchProcessor.addQuoteData()`
  - 回调数量增长，导致内存增长和事件风暴
  - 重连/重新订阅链路会放大该问题

#### I2 符号映射失败后把原始 symbol 当成标准 symbol 回退

- 位置: `src/core/01-entry/stream-receiver/services/stream-subscription-context.service.ts:178-233`
- 证据:
  - `mapSymbols()` 在异常时返回原始 `symbols` (`193-200`)
  - `resolveSymbolMappings()` 将该返回值直接作为 `standardSymbols` 继续流转 (`225-233`)
  - `StreamReceiverService.subscribeStream()` 随后使用 `standardSymbols` 做 upstream 引用计数 (`460-466`)
- 风险:
  - upstream 引用计数与退订释放建立在非标准 symbol 上
  - 同一 symbol 可能因格式差异形成多个 key
  - 与用户给定的“绿地项目、禁止兼容回退逻辑”直接冲突

#### I3 重连协调器保留兼容回退与无界 fan-out

- 位置: `src/core/01-entry/stream-receiver/services/stream-reconnect-coordinator.service.ts:375-430`
- 证据:
  - 通过 `(this.clientStateManager as any).getClientsWithHeartbeatTimeout` 和 `(this.clientStateManager as any).getClientsByProvider` 做动态能力探测
  - 对批量 client 使用 `forEach(... handleReconnection().catch(...))` fire-and-forget 触发
  - 当前 `StreamClientStateManager` 已明确提供这两个方法，无需兼容分支
- 风险:
  - 类型系统失效，未来重构时可静默退化
  - 单轮巡检可瞬时产生大量未受控 Promise
  - 与当前项目“绿地项目、禁止兼容回退逻辑”冲突

#### I4 限流依赖异常时 fail-open

- 位置: `src/core/01-entry/stream-receiver/services/stream-receiver.service.ts:292-335`
- 证据:
  - `checkConnectionRateLimit()` 在 `catch` 中直接 `return true`
  - 日志文本明确写出“故障时开放”
- 风险:
  - 一旦限流依赖抖动或异常，连接门禁完全失效
  - 对 WebSocket 入口形成可被放大的资源占用面
- 归类说明:
  - 这是安全与容量保护问题，因此归为 `Important`

#### I5 gateway 缺少 owner lease 自动查找/绑定逻辑，已有回归证据

- 风险:
