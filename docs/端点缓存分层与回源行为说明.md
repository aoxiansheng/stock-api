# 端点缓存分层与回源行为说明

更新时间：2026-03-11 11:07:44 CST

## 1. 文档目的

本文档用于统一说明项目主要对外端点的缓存分层、回源行为、并发去重策略与适用场景。

重点回答以下问题：

1. 哪些端点属于强时效。
2. 哪些端点属于弱时效。
3. 哪些端点属于超长缓存 / 同步型端点。
4. 哪些端点会在消费者请求下触发回源。
5. 哪些端点应避免进入实时回源通道。

本文档只描述当前项目实现与推荐定位，不修改代码。

## 2. 术语定义

### 2.1 强时效端点

- 面向秒级或近实时数据。
- 允许短 TTL、single-flight、调度器限速、stale fallback 等策略。
- 典型能力：报价、市场状态、交易日、实时图表读模型。

### 2.2 弱时效端点

- 面向分钟级或业务容忍延迟的数据。
- 主要通过 Smart Cache 弱时效 TTL 承接。
- 允许过期后重新回源，但无需秒级一致性。

### 2.3 超长缓存 / 同步型端点

- 不是普通 TTL 响应缓存，而是“本地版本化镜像数据集”。
- 主要模式为：后台同步、持久化快照、本地读取、增量对齐。
- 对外应尽量表现为“消费本地结果”，而不是“每次请求实时代理上游”。

### 2.4 强时效读模型端点

- 面向前端页面或业务视图的聚合读模型。
- 数据由多个来源拼装而成，例如“历史基线 + 实时增量”。
- 不适合长 TTL，也不适合直接视为同步型数据集。

### 2.5 实时流端点

- 通过 WebSocket 持续推送数据。
- 重点不是 TTL，而是订阅聚合、流缓存、断线恢复与上游订阅复用。

## 3. 总体分层结论

项目的主要对外端点可分为以下五类：

| 类别 | 典型端点 | 主要缓存/复用手段 | 是否允许请求触发回源 | 目标 |
|---|---|---|---|---|
| 强时效 TTL / 调度型 | `POST /api/v1/receiver/data` | Smart Cache 强时效 TTL、调度器、single-flight、stale fallback | 是 | 在额度受限下提供实时能力 |
| 弱时效 TTL 查询型 | `POST /api/v1/query/execute` | Smart Cache 弱时效 TTL | 是 | 提供分钟级可接受延迟查询 |
| 超长缓存 / 同步型 | `GET /api/v1/query/support-list*` | `current/meta/delta` 持久化、版本链、定时同步 | 仅在初始化或自愈时可能触发一次 | 提供本地镜像式读取 |
| 强时效读模型 | `POST /api/v1/chart/intraday-line/*` | 历史基线 + 流缓存 + cursor 增量 | 是 | 为图表提供低延迟聚合视图 |
| 实时流 | `WS /api/v1/stream-receiver/connect` | 上游订阅聚合、流缓存 Hot/Warm | 订阅时可能触发上游动作 | 多客户端共享实时推送 |

## 4. 端点级矩阵

### 4.1 总矩阵

| 端点 | 时效分类 | 主缓存载体 | 稳态是否读本地 | 请求是否可能触发回源 | 并发去重方式 | 是否适合超长缓存 |
|---|---|---|---|---|---|---|
| `POST /api/v1/receiver/data` | 强时效 | Smart Cache + 调度器 | 部分是 | 是 | single-flight + scheduler | 否 |
| `POST /api/v1/query/execute` | 弱时效 | Smart Cache | 部分是 | 是 | Smart Cache 命中复用 | 否 |
| `GET /api/v1/query/support-list/meta` | 超长缓存 / 同步型 | `meta/current` 持久化 | 是 | 仅初始化/异常时 | in-flight + 分布式锁 | 是 |
| `GET /api/v1/query/support-list` | 超长缓存 / 同步型 | `current/delta` 持久化 | 是 | 仅初始化/异常时 | in-flight + 分布式锁 | 是 |
| `POST /api/v1/chart/intraday-line/snapshot` | 强时效读模型 | 历史基线 + 流缓存 | 否 | 是 | 底层 fetch 复用 + 流缓存 | 否 |
| `POST /api/v1/chart/intraday-line/delta` | 强时效增量读模型 | cursor + 流缓存 | 是 | 通常否 | cursor 增量 + 流缓存 | 否 |
| `WS /api/v1/stream-receiver/connect` | 实时流 | Stream Cache Hot/Warm | 是 | 订阅阶段可能是 | 上游 symbol 聚合 | 否 |

### 4.2 关键判断

- `support-list` 应视为超长缓存 / 同步型端点。
- `chart-intraday` 不应视为超长缓存端点，而应视为强时效读模型端点。
- `receiver/data` 与 `query/execute` 仍然属于“请求驱动回源”的 TTL 类接口，只是时效等级不同。

## 5. `support-list`：超长缓存 / 同步型端点

### 5.1 当前实现链路

代码链路：

- `src/core/01-entry/query/controller/support-list-query.controller.ts`
- `src/core/01-entry/query/services/support-list-query.service.ts`
- `src/core/03-fetching/support-list/services/support-list-read.service.ts`
- `src/core/03-fetching/support-list/services/support-list-sync.service.ts`
- `src/core/03-fetching/support-list/services/support-list-store.service.ts`
- `src/core/03-fetching/support-list/services/support-list-sync.scheduler.ts`

### 5.2 当前缓存/存储模型

`support-list` 不是普通短 TTL 响应缓存，而是持久化版本化数据集：

1. `current` 快照：`support_list_current_{type}`
2. `meta` 元数据：`support_list_meta_{type}`
3. `delta` 增量链：`support_list_delta_{toVersion}_{type}`
4. 刷新锁：`support_list_refresh_lock_{type}`

当前默认增量保留窗口：`7` 天。

### 5.3 回源行为结论

`support-list` 的设计已经满足“消费者共享一次刷新结果，不会每个请求都打上游”的目标。

#### 稳态

- 正常情况下，读取直接消费本地 `current/meta/delta`。
- 前台消费者主要是在读取本地版本化结果。

#### 非稳态

- 若首次访问无数据，或 `meta/current` 失配，则读取前会触发 `ensureTypeInitialized`。
- 这时只允许一个请求/实例真正执行刷新。
- 其他并发请求等待该刷新完成，然后继续消费本地结果。

### 5.4 为什么应归类为超长缓存

- 面向的是清单类数据，不需要秒级变化。
- 对外提供的是 `meta / full / delta` 协议，而不是“查一次拉一次”。
- 读路径主目标是消费本地镜像，而不是实时代理上游。
- 已有定时同步任务，默认每天 `2AM` 刷新。

### 5.5 适用的缓存语言

对于 `support-list`，建议统一使用以下术语：

- 版本化同步
- 本地镜像
- 超长缓存
- 增量对齐

不建议把它表述成：

- 普通弱时效 TTL 接口
- 高频实时查询接口
- 秒级回源接口

## 6. `chart-intraday`：强时效读模型端点

### 6.1 当前实现链路

代码链路：

- `src/core/01-entry/receiver/controller/receiver-chart-intraday.controller.ts`
- `src/core/01-entry/receiver/services/receiver-chart-intraday.service.ts`
- `src/core/03-fetching/chart-intraday/services/chart-intraday-read.service.ts`
- `src/core/03-fetching/chart-intraday/services/chart-intraday-cursor.service.ts`

### 6.2 当前读模型结构

`chart-intraday` 不是一个预同步的完整数据集，而是一个面向图表渲染的即时聚合读模型：

#### Snapshot

- 先取 `get-stock-history` 作为历史基线。
- 再从流缓存读取实时 `quote:{symbol}` 点位。
- 按秒桶合并、去重、排序后输出。

#### Delta

- 依赖 `cursor`。
- 直接从流缓存里取比 `lastPointTimestamp` 更新的点位。
- 不返回全量，只返回增量。

### 6.3 为什么不适合归类为超长缓存

- 它服务的是秒级更新视图，而不是稳定版本化数据集。
- `snapshot` 本身仍然依赖历史能力回源拉基线。
- `delta` 虽然主要读流缓存，但语义仍是强时效增量而不是长期快照。
- 它更适合通过：
  - 流缓存
  - cursor 协议
  - 短时 snapshot 复用
  来降低成本，而不是通过日级/周级长缓存解决。

### 6.4 正确定位

对 `chart-intraday` 的推荐定位是：

- 强时效读模型
- 图表聚合视图
- Snapshot/Delta 协议端点
- 流优先，HTTP 兜底

而不是：

- 超长缓存端点
- 版本化镜像数据集
- support-list 类端点

## 7. 其他主要端点归类

### 7.1 `POST /api/v1/receiver/data`

推荐定位：强时效 TTL / 调度型端点。

特点：

- 面向实时或近实时数据。
- 允许 Smart Cache 强时效 TTL。
- 允许进入上游调度器进行排队、合并、限速。
- 在错误时适合叠加 `stale-if-error` / `stale fallback`。

适合能力：

- `get-stock-quote`
- `get-stock-basic-info`
- `get-market-status`
- `get-trading-days`
- `get-stock-history`（当前以 single-flight / 排队优先）
- `get-crypto-quote`（已接入调度器，merge window 与 stock quote 一致）
- `get-crypto-basic-info`（当前由 `coingecko` 提供，不走 `infoway` upstream scheduler）
- `get-crypto-history`（已接入调度器，single_symbol_only 模式）

### 7.2 `POST /api/v1/query/execute`

推荐定位：弱时效 TTL 查询型端点。

特点：

- 主要消费 Smart Cache 弱时效 TTL。
- 命中时直接返回，过期时可能回源。
- 仍属于“请求驱动回源”的查询接口，只是回源频率应低于强时效端点。

### 7.3 `WS /api/v1/stream-receiver/connect`

推荐定位：实时流端点。

特点：

- 重点不是普通 TTL，而是上游订阅聚合。
- 多个客户端应共享同一上游 symbol 订阅。
- 本地通过流缓存承接重复读取与断连恢复。

## 8. 是否允许消费者请求触发回源

### 8.1 可以接受请求触发回源的端点

- `POST /api/v1/receiver/data`
- `POST /api/v1/query/execute`
- `POST /api/v1/chart/intraday-line/snapshot`
- 部分初始化场景下的 `GET /api/v1/query/support-list*`

### 8.2 应尽量避免请求频繁触发回源的端点

- `GET /api/v1/query/support-list*`
- `POST /api/v1/chart/intraday-line/delta`
- `WS /api/v1/stream-receiver/connect` 的重复订阅

### 8.3 严格目标

对于 `support-list` 这类同步型端点，正确目标不是“绝对禁止任何请求触发回源”，而是：

1. 稳态请求应只消费本地结果。
2. 非稳态下只允许一个刷新动作回源。
3. 其他并发消费者只等待或消费刷新结果。
4. 不允许每个消费者都各自触发上游请求。

当前 `support-list` 实现已经满足上述目标。

## 9. 推荐的统一口径

后续在文档、评审、需求讨论里，建议统一使用下表：

| 端点/能力 | 推荐口径 |
|---|---|
| `support-list` | 超长缓存 / 版本化同步 / 本地镜像 |
| `chart-intraday` | 强时效读模型 / Snapshot-Delta 协议 |
| `receiver/data` | 强时效 TTL / 调度型接口 |
| `query/execute` | 弱时效 TTL 查询接口 |
| `stream-receiver` | 实时流 / 订阅聚合接口 |

## 10. 最终结论

### 10.1 `support-list`

- 是的，应视为超长缓存端点。
- 更准确地说，是“版本化同步型本地镜像端点”。
- 当前实现已经基本符合该定位。

### 10.2 `chart-intraday`

- 不应视为超长缓存端点。
- 应视为“强时效读模型端点”。
- 它应继续依赖历史基线、流缓存和 cursor 协议，而不是按 `support-list` 模式改造成同步型镜像。

### 10.3 项目级分类

当前项目的缓存/读取架构可以概括为：

- 用 Smart Cache 承接 TTL 型查询。
- 用调度器承接受额度限制的强时效回源。
- 用版本化持久化承接同步型长周期数据。
- 用流缓存与订阅聚合承接实时推送。
- 用读模型服务承接图表与视图类聚合输出。
