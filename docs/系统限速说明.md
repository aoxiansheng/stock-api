# 系统限速说明

## 1. 文档目的

本文档用于说明当前项目中与“限速 / 削峰 / 聚合”直接相关的能力，包括：

- HTTP 入口限速
- 上游 REST 请求调度限速
- WebSocket 上游订阅聚合
- WebSocket 防护规则的当前实现状态
- 已完成的压力测试结论

本文档基于当前代码实现与本地实测结果整理，不讨论未来规划项。

---

## 2. 总体结论

当前系统中，与流量控制最相关的能力可以分为三层：

1. **HTTP 入口限速**：保护本系统 API，不让请求无限制进入业务处理链路。
2. **上游 REST 调度器**：对进入上游的 REST 请求做排队、合并、降速发车。
3. **上游 WebSocket 订阅聚合**：多个客户端订阅同一 symbol 时，仅保留一份上游订阅。

这三层的职责不同：

- HTTP 入口限速：限制**客户端打我们**
- REST 调度器：限制**我们打上游**
- WS 订阅聚合：减少**重复上游订阅**

## 3. HTTP 入口限速

### 3.1 当前配置

代码位置：`src/app.module.ts`

当前实现：

```ts
ThrottlerModule.forRoot([
  {
    ttl: parseInt(process.env.AUTH_RATE_LIMIT_TTL || "60000"),
    limit: parseInt(process.env.AUTH_RATE_LIMIT_LIMIT || "100"),
  },
])
```

当前默认含义：

- 时间窗口：`60000ms`，即 **60 秒**
- 请求上限：`100`

即：

> **单个限流窗口内最多允许 100 个 HTTP 请求进入系统。**

### 3.2 触发条件

当某一限流键在 60 秒窗口中累计超过 100 次请求时，系统会直接拒绝后续请求，并返回：

- HTTP 状态码：`429`
- 异常类型：`ThrottlerException`

### 3.3 如何理解

这一层是**系统入口保护**，作用于客户端请求到本系统 API 的入口。

它限制的是：

- `POST /api/v1/receiver/data`
- `POST /api/v1/query/execute`
- 其他 HTTP API

而不是上游请求。

### 3.4 实测现象

在多次高压 REST 压测中，已明确观察到：

- 当压测请求过于密集时
- 首先触发的是本系统自己的 `ThrottlerException`
- 表现为 `POST /api/v1/receiver/data - 429`

因此，在当前默认配置下，HTTP 入口限速是系统最先触发的硬边界之一。

## 4. 上游 REST 调度限速

### 4.1 当前配置

配置位置：`.env`

当前已配置：

```env
UPSTREAM_SCHEDULER_ENABLED=true
UPSTREAM_SCHEDULER_ALLOWLIST=infoway:get-market-status,infoway:get-stock-quote,infoway:get-stock-basic-info,infoway:get-stock-history,infoway:get-crypto-quote,infoway:get-crypto-basic-info,infoway:get-crypto-history
UPSTREAM_SCHEDULER_DEFAULT_RPS=1
UPSTREAM_SCHEDULER_MAX_QUEUE_SIZE=1000
UPSTREAM_SCHEDULER_429_COOLDOWN_MS=5000
UPSTREAM_SCHEDULER_QUOTE_MERGE_WINDOW_MS=50
UPSTREAM_SCHEDULER_BASIC_INFO_MERGE_WINDOW_MS=80
```

### 4.2 当前生效能力

当前 allowlist 中，进入调度器的能力为：

- `infoway:get-market-status`
- `infoway:get-stock-quote`
- `infoway:get-stock-basic-info`
- `infoway:get-stock-history`
- `infoway:get-crypto-quote`
- `infoway:get-crypto-basic-info`
- `infoway:get-crypto-history`

### 4.3 当前速率含义

`UPSTREAM_SCHEDULER_DEFAULT_RPS=1` 的含义是：

> **同一个 `provider:capability` 队列，默认按每秒 1 次的节奏向上游发车。**

这里的“发车”是指：

- 一个调度队列取出一个已合并的任务批次
- 然后向上游发起一次真实请求

### 4.4 调度器不是入口限流器

这一层不是简单的“超过就拒绝”。

它的主要行为是：

1. 请求入队
2. 在短窗口内尝试合并兼容请求
3. 按队列顺序串行发车
4. 控制每个队列的最小发车间隔

只有在以下情况才会明确拒绝或暂停：

- 队列长度超过 `UPSTREAM_SCHEDULER_MAX_QUEUE_SIZE=1000`
- 上游返回 `429`，进入 `5000ms` 冷却期

### 4.5 `get-stock-quote` 的合并规则

对于 `get-stock-quote`，调度器的 merge key 当前只看：

- `provider`
- `capability`
- `market`
- `realtime`

**不看 `symbols`**。

这意味着：

- 用户 A 请求 `AAPL.US`
- 用户 B 请求 `QQQ.US`
- 用户 C 请求 `AAPL.US`
- 用户 D 请求 `00700.HK`

只要它们满足：

- 都是 `get-stock-quote`
- 都走同一个 provider
- `realtime` 一致
- `market` 参数兼容
- 都落在 merge window 内

它们就会被合并成一个批次，把 symbol 做并集后发往上游。

### 4.6 为什么 `get-stock-quote` 可以跨地区合并

已通过上游实测确认：

- Infoway 的 `/stock/batch_trade/{codes}`
- 支持一次请求中混合股票地区代码
- 例如：`AAPL.US,00700.HK`

因此，当前项目中的 `get-stock-quote`：

- 可以支持多用户
- 可以支持多地区股票
- 可以支持重复 symbol 合并
- 然后统一通过上游批量报价接口拉取

这是当前系统中最重要的削峰能力之一。

## 5. WebSocket 上游订阅聚合

### 5.1 当前配置

配置位置：`.env`

```env
UPSTREAM_SUBSCRIPTION_COORDINATOR_ENABLED=true
UPSTREAM_SUBSCRIPTION_UNSUBSCRIBE_GRACE_MS=15000
```

### 5.2 当前行为

这一层不是传统意义上的“限速器”，而是**上游订阅聚合器**。

它的行为是：

- 当第一个客户端订阅某个 symbol 时，系统才向上游发起真实订阅
- 当后续客户端订阅同一个 symbol 时，系统只复用已有上游订阅
- 当最后一个客户端退订时，不会立刻向上游退订，而是进入 `15000ms` 的延迟退订窗口
- 如果在这 15 秒内又有新的客户端重新订阅，会取消这次待退订动作

### 5.3 如何理解

它解决的问题是：

- 不是每个客户端都独立占用一次上游订阅额度
- 而是本系统内部做引用计数
- 多个客户端共享同一份上游订阅

这对上游 WebSocket 配额控制非常关键。

### 5.4 当前接线状态

这一能力不是“只实现未接入”，而是**已接入实际订阅链路**。

订阅时：

- 通过 `acquire()` 判断哪些 symbol 需要真正订阅上游

退订时：

- 通过 `scheduleRelease()` 控制延迟退订

因此，当前可以认定：

> **系统已经具备“多个客户端共享同一个上游 symbol 订阅”的能力。**

## 6. WebSocket 防护规则

### 6.1 当前实现的默认阈值

代码中已经实现 WebSocket 防护规则，默认值为：

- 每 IP 最大连接数：`10`
- 每用户最大连接数：`5`
- 每分钟消息数限制：`120`
- 10 秒内突发消息限制：`20`
- 每连接最大订阅数：`50`

### 6.2 这些规则的意义

这些限制主要用于防止：

- 单 IP 建立过多连接
- 同一用户建立过多连接
- 单连接在短时间内发过多消息
- 单连接订阅过多 symbol

### 6.3 当前结论边界

本轮代码审阅已确认：

- WebSocket 防护 Guard 实现存在
- 限制逻辑本身存在
- 配置值来源清晰

但本轮未把它作为“已压测确认的主要生效边界”来下结论。

因此，对 WebSocket 防护层应做如下表述：

> **这是当前已实现的 WS 防护规则；但在本轮限速结论中，最明确已实测触发的是 HTTP 入口限流，而不是这层 WS Guard。**

## 7. 当前系统如何触发限速 / 削峰

### 7.1 客户端打我们太快

触发层：HTTP 入口限流

表现：

- 返回 `429`
- 异常为 `ThrottlerException`

这是“请求还没真正进入业务逻辑前”就被拦住的情况。

### 7.2 请求进入业务后要访问上游

触发层：REST 调度器

表现：

- 请求不会立刻全部打向上游
- 会先排队
- 会在短窗口内合并
- 按每秒固定发车速率向上游发送

这是“业务放行了，但系统主动控制出站速率”的情况。

### 7.3 上游返回 429

触发层：REST 调度器冷却

表现：

- 当前队列进入冷却期
- 后续发车暂停一段时间
- 当前配置冷却时间为 `5000ms`

这是“上游明确反馈压力过大时”的保护行为。

### 7.4 多个客户端订阅同一 symbol

触发层：上游订阅聚合器

表现：

- 只保留一份上游订阅
- 其他客户端共享这份上游流
- 最后一个客户端退订后延迟 15 秒再真正退上游

这是“通过聚合减少上游配额消耗”的行为，不是传统的请求限流。

## 8. 已完成的压力测试结论

### 8.1 `get-stock-quote` 多用户多地区压测已通过

已通过新的压测脚本验证：

- 虚拟多用户
- 多地区股票（US/HK 混合）
- 用户间存在重复 symbol
- 请求会进入系统调度器合并路径

实测通过档位包括：

1. `48` 请求，`6` 并发，`500ms` 全局发包间隔，成功率 `100%`
2. `50` 请求，`8` 并发，`300ms` 全局发包间隔，成功率 `100%`
3. `90` 请求，`10` 并发，`250ms` 全局发包间隔，成功率 `100%`

第三档结果：

- 总请求数：`90`
- 成功率：`100%`
- 达成吞吐：约 `3.86 RPS`

### 8.2 结论如何理解

这并不意味着“系统整体对外无限制支持 3.86 RPS 以上所有场景”。

它说明的是：

- 在当前多用户、多地区、重复 symbol 模型下
- 通过调度器合并 + 全局节流方式
- `get-stock-quote` 这条链路可以稳定运行

同时也再次说明：

- 如果客户端直接无节制猛打 HTTP
- 最先触发的是入口 `Throttler`
- 而不是上游配额

## 9. 当前对外能力的正确理解

### 9.1 不应简单理解为“每个用户都占一份上游额度”

对于 REST `get-stock-quote`：

- 多个用户的请求会被调度器合并
- 相同或兼容条件下的不同 symbol 会一起发往上游

对于 WebSocket：

- 多个客户端订阅同一个 symbol，会共享一份上游订阅

因此：

> **当前系统的设计目标不是“每个客户端各打各的上游”，而是“先进入系统，再由系统做削峰、聚合、复用”。**

### 9.2 当前系统最需要关注的边界

如果保持默认配置不变，优先需要关注的边界是：

1. HTTP 入口限流 `100/60s`
2. 上游调度器每队列 `1 RPS`
3. 上游真实配额限制

而不是单纯盯着“在线用户数”。

## 10. 建议

### 10.1 对 REST 使用方式的建议

`get-stock-quote` 更适合作为：

- 聚合型报价接口
- 上游削峰入口
- 热门 symbol 统一调度入口

不建议把它当成：

- 大量终端用户的高频直接轮询接口

### 10.2 对大量终端实时行情的建议

如果未来要面向更多终端用户实时看盘，建议优先采用：

- WebSocket 下游广播
- 热门 symbol 共享订阅
- 内部流缓存 / 热点 fanout
- 终端少轮询、更多订阅

这样才能真正放大“少量上游额度服务大量客户端”的能力。

---

## 11. 最终结论

当前系统的限速/削峰能力可以用一句话概括：

> **客户端先受 HTTP 入口限流保护，进入系统后的上游 REST 请求再由调度器排队合并降速发车，而 WebSocket 订阅则通过 symbol 引用计数聚合，避免多个客户端重复占用上游订阅额度。**

当前最明确已验证生效的是：

- HTTP 入口限流：`100 / 60s`
- REST 调度器发车速率：每 `provider:capability` 队列约 `1 RPS`
- WebSocket 上游订阅聚合：已接入实际订阅链路
