# 分时图 release 条件退订方案单机场景完成度结论

## 2. 总结结论

在单机场景下，`分时图 release 条件退订方案` 的**主流程大体可用**；但这不等于正式方案已经全部完成。

更准确地说：

- 条件退订主流程：**单机下已跑通**
- 单机共享上游 + 多下游 session 模型：**单机下已可用**
- `snapshot / delta / release` 的 session 化改造：**已落地**
- HTTP + WS 下的基础续租链路：**已落地**
- 多实例共享状态：**仍是正式缺口**
- `release` 幂等性：**基础返回语义已进入实现，跨实例一致性仍待确认**
- 对外文档收口：**已开始同步，但不能写成“所有边界已关闭”**

## 3. 已完成项

### 3.1 两层资源模型已落地

当前代码已经把资源模型拆成两层：

- 共享上游资源：`upstreamKey`
- 下游消费会话：`sessionId`

语义上已经不是“按 symbol 直接退订”，而是“按 session 释放占用”。

### 3.2 Snapshot / Delta / Release 已完成 session 化

- `snapshot` 会创建并返回 `sessionId`
- `delta` 要求携带 `sessionId`
- `release` 也要求按 `sessionId` 释放

这说明协议主语义已经切换完成。

### 3.3 条件退订与宽限期已实现

当前行为已经符合主方案预期：

- 若仍有其他活跃 session，则不退上游
- 最后一个 session 退出后，不立即退上游
- 先进入宽限期
- 宽限期结束后才尝试真正 `unsubscribe`
- 宽限期内若重新进入，会取消待退订

### 3.4 会话所有权校验已实现

当前代码会校验：

- ownerIdentity
- symbol
- market
- provider

因此已经具备“只能释放自己的 session”的基本安全约束。

### 3.5 HTTP + WS 基础续租链路已具备

当前 session 的续租来源包括：

- HTTP `delta`
- WS `subscribe(sessionId)`
- WS `ping`

对单机场景来说，这已经足够支撑主流程跑通；但它不能替代正式方案里对多实例一致性的要求。

## 4. 当前仍是正式缺口的项

### 4.1 多实例共享状态

原方案里把多实例共享状态列为核心约束。

即使当前调试环境是单机，也不能把该项写成“当前不适用”或“已不是缺口”，原因是：

- 当前代码已经使用共享 Redis key 与跨实例相关的状态模型
- 对外协议里的 `upstreamReleased / graceExpiresAt` 一旦对前端承诺，就必须考虑多实例下的真实语义
- 该项是否关闭，只能由正式代码修复与回归测试结论决定

## 5. 本轮补齐项

### 5.1 release 幂等性已进入实现

当前行为已调整为：

- 同一个 `sessionId` 连续释放多次不报错
- 重复释放返回 `200`
- 业务层返回可体现：
  - `sessionReleased=false`
  - `reason=ALREADY_RELEASED`

补充说明：

- 以上结论只能说明“基础幂等返回语义已进入代码”
- 多实例下是否已具备共享串行化，以及 `upstreamReleased / graceExpiresAt` 是否来自共享状态，仍需最终回归确认
- 因此这里不能写成“幂等问题已彻底关闭”

### 5.2 对外文档已同步收口

当前文档已同步以下约束：

- `snapshot` 返回 `sessionId`
- `delta` / `release` 必须携带 `sessionId`
- HTTP + WS 模式下可使用 `subscribe(sessionId)` 绑定当前 Socket
- WS 稳定后若暂停高频 `delta`，需要继续发送应用层 `ping` 续租 session
- `release` 后若要继续消费，需要重新 `snapshot`

## 6. 优先级建议

在单机场景下，建议这样理解优先级：

### 已落地的部分

- session 化模型
- 条件退订
- 宽限期退订
- owner/context 校验
- 单机主流程联通

### 不能被文档抹掉的缺口

- 多实例共享状态

### 需要主 agent 结合代码最终确认的部分

- 多实例共享状态
- `release` 重复调用时 `upstreamReleased / graceExpiresAt` 的共享一致性
- 若最终以多实例部署为目标，是否已经具备可验收的 release 幂等实现

## 7. 最终判断

最终判断如下：

> 如果只看单机主流程，当前能力可以判定为“基本可用”；但如果以正式方案作为验收口径，当前还不能写成“已完成”。

必须保留的边界是：

- 多实例共享状态仍是正式缺口
- `release` 的跨实例一致性需要由主 agent 结合最终代码与回归测试确认
