# 分时图 release 幂等性处理优先级分析

更新时间：2026-03-17

## 1. 结论

2026-03-17 更新：

- 当前仓库已经引入“重复 `release(sessionId)` 返回 `ALREADY_RELEASED`”的基础语义
- 但多实例下的共享一致性，以及 `upstreamReleased / graceExpiresAt` 的精确性，仍需结合本轮代码修复与回归结果最终确认
- 因此本文不能被当成“问题已彻底关闭”的归档结论

以下内容保留为历史优先级分析背景。

`release` 幂等性当时**不属于必须立刻中断其他开发并优先修复的问题**，但也**不建议长期拖延**。

更准确的优先级判断：

- 当前优先级：`中高`
- 建议时机：下一轮分时图协议收口 / 前端联调 / 对外契约冻结前处理
- 当前是否阻断上线：**如果调用方完全可控且可接受退出清理时的 409，可暂不阻断**

## 2. 当前实现现状

当前 `release` 已经完成了“按 session 释放”的核心语义：

- 释放的是当前 `sessionId`
- 若同一 `upstreamKey` 还有其他活跃 session，不退上游
- 若已无活跃 session，则进入宽限期，宽限期后再退上游

相关实现位置：

- `src/core/03-fetching/chart-intraday/services/chart-intraday-stream-subscription.service.ts`
- `src/core/03-fetching/chart-intraday/services/chart-intraday-session.service.ts`
- `src/core/03-fetching/chart-intraday/services/chart-intraday-read.service.ts`

当前代码已经开始向幂等语义收口：

1. 第一次 `release(sessionId)` 成功后，会写入 released-session 语义记录
2. 重复 `release` 目标上会返回 `200`，并带 `reason=ALREADY_RELEASED`
3. TTL 过期后的重复 `release` 也在向“按清理成功处理”收口

但当前仍不能直接宣告“幂等问题已完全结束”，因为还存在待确认边界：

1. 多实例下的 release 共享串行化是否真正成立，仍需回归验证
2. `upstreamReleased / graceExpiresAt` 是否来自共享状态，而不是某个实例的局部观察，仍需最终确认

也就是说，当前语义是：

- “已释放” 已开始按“正常幂等清理成功”返回

而不是文档建议的：

- “已释放” 既按幂等成功返回，又在多实例下保持共享状态一致

## 3. 为什么它现在不是最高优先级

从后端资源正确性看，原始问题的第一阶段目标已经完成：

- 第一次 `release` 已能完成真正的资源释放判断
- 重复 `release` 已不再默认走 `409 SESSION_CONFLICT`
- 协议与前端文档也已开始按 `sessionId` 语义收口

当前剩余问题更像是：

- **共享状态一致性问题**
- **多实例返回值准确性问题**
- **最终验收口径问题**

而不是：

- 共享资源错误释放问题
- 数据错误问题
- 生产资源泄漏问题

如果最终部署仍是单实例，当前实现大概率足以支撑主流程；但只要准备按共享 Redis 状态运行多实例，就不能把这部分剩余边界继续视为“文档层面的次要问题”。

## 4. 为什么又不能拖太久

虽然不是最高优先级，但它在实际接入中很容易自然暴露。

### 4.1 重复释放并不是低概率边角场景

以下场景都可能自然触发第二次 `release`：

- 页面快速切换 symbol
- 组件重复卸载
- 路由切换与页面销毁顺序不稳定
- 前端清理逻辑重入
- 网络重试或客户端重放
- session 已因 TTL 过期，页面退出时仍按常规调用 `release`

### 4.2 当前 HTTP + WS 路径更容易触发“已过期再 release”

当前实现里，session 活跃续租来源是：

- HTTP `delta`
- WebSocket `subscribe(sessionId)`
- WebSocket `ping`

但不是所有 WS 活跃态都会自动续租。

如果前端进入“WS 稳定后暂停 delta”的模式，又没有稳定发送 `ping`，则 session 可能先过期；页面退出时再调用 `release`，就会收到 `409 SESSION_CONFLICT`。

### 4.3 对外契约已经开始偏离“清理型接口”的常识

`release` 本质是清理动作，调用方通常期望：

- 多调一次也没关系
- 已经清理过了也不算失败

当前返回 `409`，会让客户端不得不写额外兼容逻辑，增加接入脆弱性。

## 5. 当前风险等级判断

### 5.1 可接受的前提

在以下前提下，可以暂不立刻修改后端实现：

- 调用方数量少且完全可控
- 前端或调用方可以把 `409 SESSION_CONFLICT` 视为“已释放/已过期”的等价成功
- 当前阶段主要目标仍是验证单实例 session 模型与条件退订主流程

### 5.2 不再建议继续拖延的信号

一旦出现以下任一情况，建议尽快补齐幂等：

- 准备正式前端联调
- 准备对外发布该接口契约
- 需要补齐 `docs/done` 文档并长期维护
- 调用方不再是单一受控客户端
- 清理阶段已出现大量无意义 409 日志
- 产品/客户端明确希望退出流程“无错误感知”

## 6. 与当前文档状态的关系

当前仓库中的 done 文档仍存在协议滞后：

- `delta` / `release` 示例未体现 `sessionId`
- HTTP + WS 接入示例仍强调“WS 稳定后可暂停高频 delta”
- 但未把 `subscribe(sessionId)` / `ping` 续租约束完整表达出来

在这种文档未完全收口的状态下，如果继续保持 `release` 非幂等，会进一步增加客户端误用和误判概率。

因此，**幂等性并不一定要今天修，但应与文档收口放在同一轮处理**。

## 7. 建议决策

建议采用以下决策：

### 方案 A：现在不改代码，只做临时约束

适用条件：

- 当前仍处于内部开发或受控联调阶段

建议配套动作：

- 前端把 `409 SESSION_CONFLICT` 视为退出清理成功
- 内部约定 `release` 为 best-effort，不参与用户可见报错
- 在待办中明确记录“release 幂等化”仍未完成

优点：

- 不打断当前主线开发
- 成本最低

缺点：

- 契约语义仍不理想
- 调用方需要带额外兼容判断

### 方案 B：下一轮协议收口时一并修复

这是当前最推荐的方案。

建议与下列事项一起处理：

- `docs/done` 文档同步为 `sessionId` 版本
- HTTP + WS 续租来源说明补齐
- `release` 返回统一幂等语义

优点：

- 一次性把协议、实现、文档对齐
- 避免前端按旧文档接入后再返工

## 8. 最终建议

最终建议：

- **基础幂等返回已经进入代码，不应再回退**
- **当前优先项应转为补齐多实例共享一致性与文档口径**

一句话概括：

> 当前 `release` 已具备基础幂等返回，但还不能把“多实例下一致正确”写成既成事实；下一步重点是把共享状态与文档结论一起收口。
