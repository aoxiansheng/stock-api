# 日志模式激活说明

更新时间：2026-03-12

## 1. 目标

本文说明当前仓库中：

1. 如何切换运行模式（`dev / debug / test / prod`）；
2. 如何激活全局日志级别（`info / debug / trace`）；
3. 如何只打开指定模块的调试日志；
4. 如何激活本次分时链路相关的调试日志。

---

## 2. 先区分两个概念

当前项目里，**运行模式** 和 **日志模式** 不是同一个概念。

### 2.1 运行模式

运行模式主要由 `NODE_ENV` 和 npm script 决定，影响：

- 启动方式；
- 默认日志级别；
- 测试/生产环境行为。

### 2.2 日志模式

日志模式主要由以下环境变量决定：

- `LOG_LEVEL`
- `ENHANCED_LOGGING_ENABLED`
- `LOG_DEBUG_MODULE`

它们控制的是：

- 全局输出到什么级别；
- 是否启用增强日志控制器；
- 是否只给指定模块打开 `debug`。

---

## 3. 当前可用启动命令

`package.json` 中当前与运行模式直接相关的命令如下：

```bash
bun run dev
bun run start
bun run start:debug
bun run start:test
bun run start:prod
```

对应含义：

### 3.1 开发模式

```bash
bun run dev
```

特点：

- 使用 `bun --watch src/main.ts`；
- 适合本地日常开发；
- 若未显式设置 `NODE_ENV=production/test`，当前 logger 默认级别通常是 `debug`。

### 3.2 调试器模式

```bash
bun run start:debug
```

特点：

- 启动 Node Inspector；
- 适合断点调试；
- **注意：它表示“调试器模式”，不等于日志自动进入 debug 模式**；
- 如果要同时看 debug 日志，仍建议显式加上 `LOG_LEVEL=debug`。

### 3.3 测试模式

```bash
bun run start:test
```

特点：

- 会设置 `NODE_ENV=test`；
- 当前默认日志级别会变成 `warn`；
- 如果测试环境也要看 debug，需要显式设置 `LOG_LEVEL=debug`。

### 3.4 生产模式

```bash
NODE_ENV=production bun run start:prod
```

特点：

- 当前默认日志级别是 `info`；
- 一般不建议直接开全局 `debug`；
- 若需要排障，优先使用“模块级 debug”。

---

## 4. 默认日志级别规则

当前实现的默认规则是：

- `NODE_ENV=production` → 默认 `info`
- `NODE_ENV=test` → 默认 `warn`
- 其他环境（如开发环境）→ 默认 `debug`

也就是说：

### 4.1 本地开发时

如果你直接：

```bash
bun run dev
```

通常已经接近 debug 输出。

### 4.2 测试和生产时

如果你不额外设置：

- `test` 更偏保守；
- `prod` 默认不输出 `debug`。

---

## 5. 如何激活全局日志级别

### 5.1 全局 debug

```bash
LOG_LEVEL=debug bun run dev
```

或：

```bash
LOG_LEVEL=debug bun run start
```

适用场景：

- 本地排查多模块协作问题；
- 需要整体看 debug 日志；
- 不在意日志量较大。

### 5.2 全局 trace

```bash
LOG_LEVEL=trace bun run dev
```

适用场景：

- 极细粒度排障；
- 临时使用，不建议长期常开。

### 5.3 全局 info / warn / error

```bash
LOG_LEVEL=info bun run dev
LOG_LEVEL=warn bun run dev
LOG_LEVEL=error bun run dev
```

适用场景：

- 主动降噪；
- 本地模拟生产日志量。

---

## 6. 如何激活增强日志控制

当前项目支持增强日志控制器。

### 6.1 启用方式

```bash
ENHANCED_LOGGING_ENABLED=true
```

### 6.2 作用

启用后：

- 日志是否输出，会先走统一日志控制器判断；
- 支持模块级别覆盖；
- 支持 `LOG_DEBUG_MODULE` 仅打开指定模块的 `debug`。

### 6.3 不启用时的行为

如果不设置或设置为 `false`：

- `shouldLog(...)` 当前会直接放行；
- 此时真正能否看到 `debug`，主要仍由底层 `LOG_LEVEL` 决定；
- 也就是说：
  - `LOG_LEVEL=debug` + 未启用增强日志 = **大多数 debug 都会输出**；
  - 不能做到“只开指定模块 debug”。

因此：

**如果你的目标是“只看某几个模块的 debug”，必须启用 `ENHANCED_LOGGING_ENABLED=true`。**

---

## 7. 如何只打开指定模块的 debug

### 7.1 基本写法

```bash
ENHANCED_LOGGING_ENABLED=true LOG_LEVEL=debug LOG_DEBUG_MODULE=ModuleA,ModuleB bun run dev
```

说明：

- `ENHANCED_LOGGING_ENABLED=true`：启用增强日志控制；
- `LOG_LEVEL=debug`：允许底层实际输出 debug；
- `LOG_DEBUG_MODULE=...`：只把指定模块提升到 debug。

### 7.2 为什么还需要 `LOG_LEVEL=debug`

因为模块级控制只解决“是否允许记这条日志”；
底层 logger 本身还需要全局允许输出 `debug`。

所以推荐组合永远是：

```bash
ENHANCED_LOGGING_ENABLED=true LOG_LEVEL=debug LOG_DEBUG_MODULE=... bun run dev
```

而不是只写：

```bash
LOG_DEBUG_MODULE=...
```

## 8. 当前分时链路推荐激活方式

这次分时诊断相关日志已经收敛到统一 debug 体系，相关模块为：

- `StreamDataProcessor`
- `ChartIntradayReadService`
- `StreamCacheStandardizedService`

### 8.1 推荐命令

```bash
ENHANCED_LOGGING_ENABLED=true LOG_LEVEL=debug LOG_DEBUG_MODULE=StreamDataProcessor,ChartIntradayReadService,StreamCacheStandardizedService bun run dev
```

### 8.2 适用场景

适合排查：

- 分时流入口是否进入处理链；
- 标准化后的 symbol 是否一致；
- StreamCache 是否命中/未命中；
- snapshot / delta 是否正确读取流缓存；
- cursor 边界与实时点位是否对齐。

## 9. 常见使用模板

### 9.1 本地普通开发

```bash
bun run dev
```

### 9.2 本地全局 debug

```bash
LOG_LEVEL=debug bun run dev
```

### 9.3 本地只开分时链路 debug

```bash
ENHANCED_LOGGING_ENABLED=true LOG_LEVEL=debug LOG_DEBUG_MODULE=StreamDataProcessor,ChartIntradayReadService,StreamCacheStandardizedService bun run dev
```

### 9.4 测试环境临时看 debug

```bash
NODE_ENV=test LOG_LEVEL=debug bun run start:test
```

### 9.5 生产环境临时排障（谨慎使用）

```bash
NODE_ENV=production ENHANCED_LOGGING_ENABLED=true LOG_LEVEL=debug LOG_DEBUG_MODULE=StreamDataProcessor,ChartIntradayReadService,StreamCacheStandardizedService bun run start:prod
```

建议：

- 仅短时使用；
- 排障结束立即恢复；
- 优先打开指定模块，不要长期开全局 debug。

## 10. `.env.local` 推荐写法

如果你不想每次手工拼环境变量，可以临时写入 `.env.local`：

### 10.1 本地全局 debug

```env
LOG_LEVEL=debug
```

### 10.2 本地只开分时链路 debug

```env
ENHANCED_LOGGING_ENABLED=true
LOG_LEVEL=debug
LOG_DEBUG_MODULE=StreamDataProcessor,ChartIntradayReadService,StreamCacheStandardizedService
```

当前配置模块会优先读取：

1. `.env.local`
2. `.env`

因此建议：

- 本地临时调试放 `.env.local`
- 通用配置才考虑 `.env`

## 11. 排障建议

### 11.1 如果发现 debug 没打印

按顺序检查：

1. 是否已经设置 `LOG_LEVEL=debug`；
2. 是否需要模块级 debug；
3. 如果需要模块级 debug，是否设置了 `ENHANCED_LOGGING_ENABLED=true`；
4. `LOG_DEBUG_MODULE` 中模块名是否与代码里的 logger context 完全一致。

### 11.2 当前分时链路对应的模块名必须完全一致

可直接使用：

```text
StreamDataProcessor
ChartIntradayReadService
StreamCacheStandardizedService
```

### 11.3 如果日志过多

优先收缩成模块级调试，而不是直接把全局 `LOG_LEVEL=debug` 改回 `info`。

推荐方式：

```bash
ENHANCED_LOGGING_ENABLED=true LOG_LEVEL=debug LOG_DEBUG_MODULE=ChartIntradayReadService bun run dev
```

只观察一个模块，最容易定位问题。

---

## 12. 当前建议

### 推荐默认策略

- 开发阶段：按需使用 `LOG_LEVEL=debug`
- 日常开发：默认 `bun run dev`
- 定位特定链路：使用模块级 debug
- 生产排障：短时、定向、可回滚

### 推荐排障策略

如果只排查分时链路，优先使用：

```bash
ENHANCED_LOGGING_ENABLED=true LOG_LEVEL=debug LOG_DEBUG_MODULE=StreamDataProcessor,ChartIntradayReadService,StreamCacheStandardizedService bun run dev
```

而不是：

```bash
LOG_LEVEL=debug bun run dev
```

原因是：

- 前者噪音更小；
- 边界更清晰；
- 更符合当前统一日志体系的设计目标。
