# 分时图非交易时段上游阻断与会话恢复开发方案

更新时间：2026-03-16

## 1. 背景

当前分时图链路已经具备：

- `snapshot` 首屏组装
- `delta` 增量兜底
- 内部自动上游实时订阅
- `release` 主动释放内部订阅

但当前实现存在一个核心问题：

- 只要下游请求 `snapshot` / `delta`，系统就会无条件尝试确保上游实时订阅存在
- `snapshot` 还会继续对上游发起历史基线与参考价拉取
- 在 `CN/HK` 午休、收盘后、周末、节假日，这些动作大多没有业务价值
- 若下游持续请求，会导致上游连接、心跳、REST 配额被空耗

当前问题在代码中的直接体现：

- [chart-intraday-read.service.ts](/Users/honor/Documents/code/newstockapi/stock-api/src/core/03-fetching/chart-intraday/services/chart-intraday-read.service.ts)
  - `getSnapshot()` 当前无条件执行 `ensureRealtimeSubscription(...)`
  - `getDelta()` 当前无条件执行 `ensureRealtimeSubscription(...)`
  - `getSnapshot()` 当前无条件对上游执行历史基线与参考价拉取
- [chart-intraday-stream-subscription.service.ts](/Users/honor/Documents/code/newstockapi/stock-api/src/core/03-fetching/chart-intraday/services/chart-intraday-stream-subscription.service.ts)
  - 当前 lease 每次请求都会续命
  - 只要下游持续访问，非交易时段的上游订阅就可能一直不释放

## 2. 已对齐的业务结论

本方案以以下结论为前提：

1. 分时图 API 不需要在非交易时段拒绝服务。
2. 非交易时段仍可对下游提供查询服务，但应阻断系统自身对上游的新请求。
3. 需要区分两层连接：
   - 下游用户和我们的连接
   - 我们和上游的连接
4. 下游连接可以保持，上游连接可以在非交易时段主动断开。
5. `CN/HK` 午休时，如果下游连接仍然活跃，系统应在下一个交易时段开始时自动恢复上游订阅。
6. 恢复上游订阅的前提不是“市场开盘了”，而是：
   - 市场进入可产生新数据的 session
   - 并且该标的仍然存在活跃下游消费者
7. `CRYPTO` 是例外，保持 24/7 `live` 模式。

## 3. 目标与非目标

目标：

- 在非交易时段阻断分时图对上游的新增 REST/WS 请求。
- 保留下游会话，不强制下游重连。
- 在 `CN/HK` 午休结束后，对仍有活跃下游消费者的标的自动恢复上游订阅。
- 为分时图引入 `live / paused / frozen` 三态，而不是全年统一实时模式。
- 使用现有缓存体系承接非交易时段的返回，不重复设计第二套缓存。

非目标：

- 不改造通用 `receiver/data` 的所有能力。
- 不在本方案中重构通用 `stream-receiver` 架构。
- 不在第一阶段处理 `US` 盘前盘后完整产品语义。
- 不在第一阶段承诺“所有 symbol 在非交易时段一定有首屏快照”。

## 4. 核心问题定义

当前真正的问题不是“分时图 API 是否全年可访问”，而是：

- 系统把“下游有人查”错误等价成了“必须建立上游实时链路”

正确的语义应是：

- 下游可查：表示系统应该返回一个合法响应
- 上游可连：表示当前确实存在新数据推进的必要

因此，分时图必须从“全年统一实时装配”改为“按市场 session 切换运行模式”。

## 5. 核心设计

### 5.1 三态模型

建议为分时图引入三种运行状态：

1. `live`

- 当前 market session 会产生新分时点
- 允许拉上游历史
- 允许建立或维持上游实时订阅
- `snapshot` / `delta` 都可推进图表

2. `paused`

- 仅用于“当前交易日尚未结束，但当前时段不会产生新点”的场景
- 典型场景：`CN/HK` 午休
- 下游会话继续保留
- 上游实时订阅主动释放
- 不允许新的上游 REST 拉取
- 等待下一个 `nextSessionStart` 自动恢复

3. `frozen`

- 当前请求不应再依赖上游
- 典型场景：
  - 收盘后
  - 周末 / 节假日
  - 请求的 `tradingDay` 不是市场当前交易日
  - `US` 盘前 / 盘后（第一阶段按 frozen 处理）
- 只允许返回本地冻结快照或缓存结果
- 不允许新的上游 REST/WS 请求

### 5.2 两层资源模型

本方案要求显式区分：

1. 下游消费会话 `consumerSession`

- 表示某个下游页面或连接仍在消费这个分时图
- 重点表达“用户还在不在”

2. 上游运行态 `upstreamRuntime`

- 表示我们是否真的维持了对上游的订阅
- 重点表达“当前要不要连上游”

设计原则：

- `consumerSession` 存在，不代表 `upstreamRuntime` 一定存在
- 午休时可以保留 `consumerSession`，同时释放 `upstreamRuntime`
- 到下个可交易 session，再根据 `consumerSession` 是否仍存在决定是否恢复 `upstreamRuntime`

### 5.3 Session 恢复原则

自动恢复上游订阅必须满足：

1. 市场已进入允许实时推进的 session
2. 该 `symbol + provider + wsCapabilityType` 下仍有活跃下游消费者
3. 当前消费会话仍属于同一个 `tradingDay`

不满足上述条件时，不允许因为“市场开盘了”而自动拉起上游订阅。

## 6. 市场策略矩阵

### 6.1 第一阶段策略

`CN/HK`

- `TRADING` -> `live`
- `LUNCH_BREAK` -> `paused`
- `PRE_MARKET` -> `frozen`
- `AFTER_HOURS / MARKET_CLOSED / WEEKEND / HOLIDAY` -> `frozen`

`US`

- 第一阶段仅 `TRADING` -> `live`
- `PRE_MARKET / AFTER_HOURS / MARKET_CLOSED / WEEKEND / HOLIDAY` -> `frozen`

`CRYPTO`

- 永远 `live`

### 6.2 tradingDay 规则

无论市场状态如何，只要：

- 请求的 `tradingDay` 不是当前市场交易日

则直接进入 `frozen`，不允许为该请求建立上游实时链路。

这条规则可以避免：

- 用户查看历史交易日时仍然误拉上游实时
- 隔夜页面还在用前一天会话自动恢复订阅

## 7. 请求路径改造方案

### 7.1 Snapshot

建议 `snapshot` 流程改为：

1. 解析 `symbol / market / tradingDay / provider`
2. 通过 `MarketStatusService` 计算当前 session
3. 由 `ChartIntradaySessionPolicyService` 决定本次模式：`live / paused / frozen`
4. 按模式走不同分支

`live` 分支：

- 允许确保上游实时订阅
- 允许拉历史基线
- 允许拉参考价
- 组装完成后写入冻结快照缓存

`paused / frozen` 分支：

- 禁止执行 `ensureRealtimeSubscription`
- 禁止执行新的上游历史基线拉取
- 禁止执行新的参考价拉取
- 只读取：
  - 冻结快照缓存
  - 参考价缓存
  - 已有 `stream-cache` 中仍保留的历史点位窗口

### 7.2 Delta

建议 `delta` 流程改为：

`live` 分支：

- 可确保上游实时订阅存在
- 正常从 `stream-cache` 读取并返回增量

`paused / frozen` 分支：

- 禁止执行 `ensureRealtimeSubscription`
- 不发起上游任何新请求
- 直接返回空增量：
  - `points = []`
  - `hasMore = false`
  - `nextCursor` 正常推进
  - `lastPointTimestamp` 维持原值

这样前端在午休期间可以保持轮询或连接，但不会继续驱动上游。

### 7.3 Release / Disconnect / Unsubscribe

建议与 [分时图release条件退订方案.md](/Users/honor/Documents/code/newstockapi/stock-api/docs/todo/分时图release条件退订方案.md) 联动：

- `release` 释放的是消费者会话
- 非最后一个消费者退出时，不退上游
- 最后一个消费者退出时：
  - 取消恢复计划
  - 释放上游订阅
  - 清理会话记录

### 7.4 午休恢复

对于 `CN/HK`：

- 进入 `LUNCH_BREAK` 时：
  - 保留 `consumerSession`
  - 释放 `upstreamRuntime`
  - 记录 `nextSessionStart`
- 到达 `nextSessionStart` 时：
  - 重新检查市场状态
  - 重新检查该标的是否仍有活跃下游消费者
  - 若两者都成立，恢复上游订阅
  - 若不成立，则不恢复

## 8. 缓存与状态存储设计

### 8.1 复用现有缓存器

建议直接复用 `BasicCacheService` / Redis，不单独引入新的存储组件。

原因：

- 当前系统已有统一缓存器
- 分时图运行态不是长期业务数据，不需要 MongoDB
- 会话态、冻结快照、恢复计划都更适合短 TTL 缓存

### 8.2 建议缓存键

冻结快照：

- `chart-intraday:frozen-snapshot:v1:{provider}:{market}:{symbol}:{tradingDay}`

消费者会话：

- `chart-intraday:consumer-session:v1:{sessionId}`

上游运行态：

- `chart-intraday:upstream-runtime:v1:{provider}:{wsCapabilityType}:{symbol}`

恢复计划：

- `chart-intraday:resume-ticket:v1:{provider}:{wsCapabilityType}:{symbol}`

### 8.3 TTL 建议

冻结快照：

- 建议至少覆盖当前交易日结束后的一段可查询窗口
- 第一阶段建议：`24h ~ 72h`

消费者会话：

- 由下游心跳 / delta / WS 活动持续续租
- 第一阶段建议：`2min ~ 10min`

恢复计划：

- 保持到 `nextSessionStart + buffer`
- 午休场景一般只需覆盖 1 小时量级

## 10. 分阶段实施建议

### 阶段 A：止血版

目标：

- 先消除非交易时段的无意义上游订阅

实施内容：

- `snapshot / delta` 先引入 session policy
- 非交易时段不再调用 `ensureRealtimeSubscription`
- `delta` 在非交易时段直接返回空增量

收益：

- 先解决最浪费的上游 WS 保活问题
- 改动最小，风险最低

### 阶段 B：午休暂停与恢复

目标：

- 解决 `CN/HK` 午休时“不断上游，13:00 无法恢复”的问题

实施内容：

- 引入消费者会话注册
- 午休时 pause upstream
- 到 `nextSessionStart` 时按活跃消费者自动恢复

收益：

- 满足“下游连接不掉，上游连接可断，开盘自动续上游”的核心业务诉求

### 阶段 C：冻结快照增强

目标：

- 让收盘后和非交易时段查询更加稳定，不依赖上游

实施内容：

- `snapshot` 成功后固化冻结快照
- `paused / frozen` 优先返回冻结快照
- 对参考价缓存与冻结快照统一整理

收益：

- 非交易时段真正做到“继续服务但不打上游”

## 12. 风险与边界

### 12.1 冻结快照缺失

第一阶段最现实的边界是：

- 若非交易时段首次请求某个 symbol
- 且本地没有该 `tradingDay` 的冻结快照

则严格模式下只能返回“当前无可用冻结快照”，而不能再去打上游。

这不是 bug，而是“阻断上游”的自然边界。

### 12.2 多实例一致性

如果服务是多实例部署：

- 消费者会话和恢复计划不能只放内存
- 必须以 Redis 为事实来源

本地 timer 可以存在，但必须在恢复前再次校验 Redis 状态，避免重复恢复。

### 12.3 交易日边界

恢复逻辑必须绑定 `tradingDay`：

- 午休恢复是“同一交易日内的下一 session”
- 不能让昨天留下的会话跨日自动恢复到今天

## 13. 推荐落地顺序

建议按以下顺序推进：

1. 先做 `session policy`
2. 再做非交易时段阻断上游订阅
3. 再做午休 `pause/resume`
4. 最后补冻结快照缓存与非交易时段稳定返回

原因：

- 先止损，先把最明显的上游空耗砍掉
- 再处理 `CN/HK` 午休恢复
- 最后再提高“非交易时段返回稳定性”

## 14. 建议开关

建议增加特性开关，便于灰度：

- `CHART_INTRADAY_ENABLE_SESSION_POLICY`
- `CHART_INTRADAY_ENABLE_UPSTREAM_PAUSE_RESUME`
- `CHART_INTRADAY_ENABLE_FROZEN_SNAPSHOT`

默认建议：

- 开发环境逐步开启
- 生产环境按阶段灰度

## 15. 结论

本方案的核心不是“非交易时段拒绝分时图服务”，而是：

- 对下游继续服务
- 对上游按 session 收敛
- 把“消费者会话”和“上游运行态”拆开建模

对于 `CN/HK`：

- 午休时保留下游连接
- 主动释放上游连接
- 到下一个交易时段开始时，若消费者仍在线，再恢复上游连接

这能同时满足：

- 资源节流
- 下游体验连续
- 午休恢复自动化
- 与现有架构最小冲突
