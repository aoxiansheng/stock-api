# 分时诊断开关与流客户端状态边界问题分析（2026-03-12）

## 1. 背景

在对比分支差异时，发现当前工作区存在两类值得尽快收敛的问题：

1. `CHART_INTRADAY_DIAGNOSTIC_LOG` 的读取方式不统一；
2. `StreamDataFetcherService` 在连接/订阅执行层尝试维护客户端订阅状态，职责边界不清。

这两类问题已完成收敛：第一类通过统一复用全局日志分级能力解决，第二类通过移除 Fetcher 层越界状态写入解决。

---

## 2. 结论摘要

### 2.1 分时诊断开关问题（已解决）

该问题已通过统一复用全局日志分级能力解决。历史上同一个配置项曾被三种代码路径以不同方式读取：

- `stream-data-processor` 使用 `ConfigService`；
- `chart-intraday-read` 直接读取 `process.env`；
- `stream-cache-standardized` 直接读取 `process.env`。

该问题的本质不是“功能立即错误”，而是**配置入口散落、解析逻辑重复、测试注入方式不一致**。

**最终修复结果：**三条链路已统一改为复用 `shouldLog(..., "debug")` 的全局日志分级入口，不再散落读取 `process.env` 或重复解析布尔值。

### 2.2 流客户端状态边界问题（已解决）

该问题已在本次修复中解决：`StreamDataFetcherService` 不再调用 `StreamClientStateManager` 维护客户端订阅状态。历史问题是：`StreamDataFetcherService` 只有 `connection` 和 `symbols` 上下文，却尝试调用 `StreamClientStateManager` 维护“客户端订阅状态”。

由于 `clientStateManager` 的主键语义是 `clientId`，而 `StreamDataFetcherService` 拿到的是 `connection.id`，两者并不等价，因此新增的 guard 只能起到“避免误写”的作用，**不能从根上解决职责错位**。

**最终修复结果：**已把客户端订阅状态维护完全保留在 `stream-receiver` 编排层，并移除 `StreamDataFetcherService` 内部 3 处相关副作用写入（订阅成功、取消订阅、连接关闭）。

---

## 3. 问题一：`CHART_INTRADAY_DIAGNOSTIC_LOG` 读取不统一

## 3.1 现状证据

当前该开关在以下位置被读取：

1. `src/core/01-entry/stream-receiver/services/stream-data-processor.service.ts`
- 通过 `ConfigService` + `readBooleanConfig(...)` 读取。

2. `src/core/03-fetching/chart-intraday/services/chart-intraday-read.service.ts`
- 通过 `process.env.CHART_INTRADAY_DIAGNOSTIC_LOG` + `parseBooleanEnv(...)` 读取。

3. `src/core/05-caching/module/stream-cache/services/stream-cache-standardized.service.ts`
- 通过 `process.env.CHART_INTRADAY_DIAGNOSTIC_LOG` + `parseBooleanEnv(...)` 读取。

从架构上看，这是同一个领域开关，却被不同服务各自解析。

## 3.2 当前风险

### 3.2.1 解析逻辑重复

布尔配置解析逻辑在多个类中重复实现，后续如果要支持更多取值形式，容易出现行为分叉。

### 3.2.2 测试注入方式不一致

使用 `ConfigService` 的类可以通过 Nest 注入配置进行测试；直接依赖 `process.env` 的类则更依赖全局环境，测试隔离性更差。

### 3.2.3 配置职责散落

业务服务同时承担了“业务逻辑”和“配置解析”两种职责，不利于后续维护，也不利于统一配置文档化。

## 3.3 根因判断

根因不是功能设计复杂，而是**为快速补诊断日志，直接在局部服务中就地读取环境变量**，没有把该开关收敛到统一配置入口。

这是一个典型的“局部可用、整体发散”的演化结果。

## 3.4 推荐解决方案

### 方案 A（推荐）：新增专用配置服务

新增一个很薄的配置服务，例如：

- `IntradayDiagnosticsConfigService`

职责仅包含：

- 统一读取 `CHART_INTRADAY_DIAGNOSTIC_LOG`
- 统一解析布尔值
- 对外暴露 `isEnabled(): boolean`

然后由以下服务统一注入：

- `StreamDataProcessorService`
- `ChartIntradayReadService`
- `StreamCacheStandardizedService`

### 方案 A 的优点

1. **KISS**：只抽一层，不引入额外复杂配置体系；
2. **DRY**：解析逻辑、默认值、配置 key 都只保留一份；
3. **SOLID / 单一职责**：业务服务只关心“是否开启诊断”，不关心“如何解析环境变量”；
4. **测试友好**：可直接 mock `isEnabled()`，不必在单测里频繁污染 `process.env`。

### 方案 B（次优）：全部改为注入 `ConfigService`

如果短期不想新增配置服务，则至少应统一改为注入 `ConfigService`，不要继续直接读取 `process.env`。

该方案优于现状，但仍会留下多处近似布尔解析逻辑，长期仍不如方案 A 干净。

## 3.5 不建议的方案

1. 保留当前混合模式；
2. 继续在不同类里各自维护 `parseBooleanEnv(...)`；
3. 将该开关硬塞入与其语义不完全匹配的大配置服务中，导致配置归属继续模糊。

## 3.6 建议验收标准

1. `CHART_INTRADAY_DIAGNOSTIC_LOG` 只保留一个统一读取入口；
2. 三条链路的日志开关行为一致：
   - 流入口写缓存；
   - StreamCache 读写；
   - snapshot / delta 读取；
3. 单元测试覆盖以下输入：
   - `undefined`
   - `true`
   - `1`
   - `yes`
   - `on`
   - `false`
   - `0`

## 4. 问题二：`StreamDataFetcherService` 越界维护客户端状态（已修复）

## 4.1 现状证据

当前 `StreamDataFetcherService` 在以下阶段尝试写客户端状态：

1. 订阅成功后更新状态；
2. 取消订阅后更新状态；
3. 连接关闭时移除状态。

但其可用上下文只有：

- `connection`
- `symbols`

其中被使用的是 `connection.id`。

与此同时，`StreamClientStateManager` 的主键语义明显是 `clientId`，不是 `connection.id`。

而在真正的入口编排层 `StreamReceiverService` 中，系统已经在使用真实 `clientId` 来维护客户端订阅：

- 订阅成功后 `addClientSubscription(clientId, ...)`
- 取消订阅时 `removeClientSubscription(clientId, ...)`
- 重连恢复时再次基于 `clientId` 恢复状态

这说明客户端状态的归属本来就在 `stream-receiver`，不在 `stream-data-fetcher`。

## 4.2 当前 guard 的真实含义

当前新增的 guard 逻辑大意是：

- 如果 `getClientSubscription(connection.id)` 查得到，就更新；
- 查不到，就跳过。

这可以避免把错误的 `connection.id` 盲目当作 `clientId` 写入状态，但它只是一个“止血”动作，不是根治方案。

换句话说：

**guard 修复的是症状，未修复设计边界。**

## 4.3 根因判断

根因是职责下沉过度：

- `StreamDataFetcherService` 本应是“连接与订阅执行层”；
- `StreamReceiverService` 才是“客户端上下文编排层”。

当前 fetcher 层试图顺手维护客户端状态，导致它需要知道 `clientId` 语义；但它实际上并没有稳定的 `clientId` 上下文。

这属于典型的职责越界。

## 4.4 为什么不能依赖 `connection.id == clientId`

从现有代码结构看，不应假设 `connection.id` 与 `clientId` 等价，原因包括：

1. 主入口 `stream-receiver` 在调用 `clientStateManager` 时始终使用显式 `clientId`；
2. 上游订阅协调器 `UpstreamSymbolSubscriptionCoordinatorService` 是按“客户端引用计数”运作的，说明系统允许多个客户端共享上游连接/订阅；
3. 一旦连接可复用，`connection.id -> clientId` 就不是天然一对一关系。

因此，把 `connection.id` 当 `clientId` 使用，只能是偶然成立，不能作为稳定契约。

## 4.5 推荐解决方案

### 方案 A（推荐）：移除 fetcher 层对客户端状态的写入

将 `StreamDataFetcherService` 中以下副作用移除：

1. `subscribeToSymbols(...)` 内的 `updateSubscriptionState(...)`
2. `unsubscribeFromSymbols(...)` 内的 `updateSubscriptionState(...)`
3. `closeConnection(...)` 内的 `removeConnection(...)`

保留 `StreamReceiverService` 作为唯一客户端状态写入入口。

### 方案 A 的优点

1. **单一职责清晰**：
   - fetcher 只管上游连接与订阅执行；
   - receiver 只管客户端订阅编排；
2. **避免重复写状态**：减少订阅成功后“双写状态”的隐患；
3. **避免错误主键**：不再需要把 `connection.id` 冒充 `clientId`；
4. **更符合现有代码事实**：因为当前真实调用者也只有 `stream-receiver`。

### 方案 B（不推荐但可行）：显式把 `clientId` 透传到 fetcher 层

如果业务坚持让 fetcher 层同时维护客户端状态，则必须：

- 修改接口签名；
- 显式传入 `clientId`；
- 或在 `StreamConnection` metadata 中稳定挂载 `clientId`。

但这会使 fetcher 层从“连接执行层”膨胀为“客户端编排层”，接口语义变脏，不符合当前架构分层。

因此仅作为备选，不建议采用。

## 4.6 对当前 guard 的处理建议

短期上，guard 比“无脑写入”更安全；
中长期上，应直接删除这段 guard 对应的整块客户端状态写入逻辑，而不是继续强化 guard。

换言之：

- **短期**：guard 可保留，防止误写；
- **正式修复**：移除 fetcher 层客户端状态副作用；
- **最终目标**：`StreamClientStateManager` 仅由编排层基于真实 `clientId` 驱动。

## 4.7 建议验收标准

1. `StreamDataFetcherService` 不再直接写 `StreamClientStateManager`；
2. `StreamReceiverService` 成为唯一客户端状态写入口；
3. 订阅 / 退订 / 重连恢复 / 连接关闭场景下，客户端状态一致；
4. `UpstreamSymbolSubscriptionCoordinatorService` 的引用计数行为不被破坏；
5. 不再出现依赖 `connection.id == clientId` 的隐式假设。

---

## 5. 最终建议

针对这两个问题，建议采取以下最小闭环方案：

1. 统一复用现有全局日志分级能力，收敛分时诊断日志输出入口；
2. 移除 `StreamDataFetcherService` 中对 `StreamClientStateManager` 的写操作；
3. 保持 `StreamReceiverService` 作为唯一客户端状态编排入口；
4. 进一步让 `StreamReceiverService` 直接注入 `StreamClientStateManager`，移除 Fetcher 中转 getter；
5. 为上述调整补最小可行单元测试。

该方案同时满足：

- **KISS**：不做大规模重构；
- **YAGNI**：不引入未被当前问题证明必要的映射层；
- **DRY**：统一配置解析，减少重复状态写入；
- **SOLID**：明确配置职责与客户端状态职责边界。

---

## 6. 后续实施顺序建议

建议按以下顺序落地：

1. 先收敛诊断开关配置入口；
2. 再移除 fetcher 层客户端状态副作用；
3. 最后补单测并回归：
   - 订阅/退订；
   - 连接关闭；
   - 重连恢复；
   - 分时 snapshot / delta 诊断日志链路。

这样可以把影响面拆小，便于定位回归问题。


## 补充说明（落地实现）

更新时间：2026-03-12 20:56:04 CST

### 实际落地与原建议的差异

#### 1. 分时诊断开关

最终没有新增 `IntradayDiagnosticsConfigService`，而是统一复用了现有全局日志分级能力：

- `StreamDataProcessorService`
- `ChartIntradayReadService`
- `StreamCacheStandardizedService`

三条链路统一改为通过 `shouldLog(..., "debug")` 判断是否输出分时诊断日志。

这样做的原因是：

1. 当前需求本质是“是否输出 debug 级诊断日志”，与现有日志分级模型天然一致；
2. 继续复用统一日志组件，比再引入一个薄配置服务更符合 KISS / YAGNI；
3. 避免同一语义同时存在“日志级别控制”和“独立诊断开关”两套入口。

因此，文档第 3 节中的“新增专用配置服务”保留为历史评估方案，但**实际落地方案**是：

- 统一收敛到现有日志组件；
- 不再散落读取 `process.env.CHART_INTRADAY_DIAGNOSTIC_LOG`；
- 不额外引入新的配置服务层。

#### 2. 流客户端状态边界

本问题最终分两步完成修复：

1. **第一步**：移除 `StreamDataFetcherService` 中对 `StreamClientStateManager` 的 3 处副作用写入；
2. **第二步**：进一步移除 `StreamDataFetcherService` 中转暴露 `getClientStateManager()` 的耦合，让 `StreamReceiverService` 直接注入 `StreamClientStateManager`。

最终职责边界为：

- `StreamDataFetcherService`：
  - 负责连接建立、关闭、订阅、退订、连接池与流缓存协作；
  - 不再负责客户端状态写入，也不再作为客户端状态访问入口。
- `StreamReceiverService`：
  - 作为唯一的客户端状态编排层；
  - 基于真实 `clientId` 调用 `StreamClientStateManager`。

这比仅移除副作用写入更完整，因为它同时消除了“Fetcher 作为状态管理中转入口”的残余耦合。

### 最终验收结果

1. 分时诊断日志统一复用全局日志分级能力；
2. `StreamDataFetcherService` 不再直接写客户端状态；
3. `StreamReceiverService` 成为唯一客户端状态写入口；
4. `StreamReceiverService` 直接注入 `StreamClientStateManager`，不再经由 Fetcher 中转；
5. 相关定向单测已通过。
