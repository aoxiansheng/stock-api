# 上游请求削峰与流订阅聚合方案

## 1. 背景

当前项目已经具备较完整的缓存能力，包括：

- Smart Cache：业务响应缓存
- Basic Cache：基础 Redis 缓存
- Symbol Mapper Cache：符号映射缓存
- Stream Cache：流式热/温缓存

但对于“上游供应商额度非常紧”的场景，当前仍有两个核心风险：

1. **REST 上游请求仍可能在不同 key 并发下同时打到上游**
2. **WebSocket 客户端订阅虽然已经复用连接，但还没有做到真正的上游 symbol 订阅引用计数聚合**

上游当前只提供 **2 次/秒** 的请求额度，因此系统需要从“有缓存”进一步升级为“**缓存 + 总量闸门 + 请求排序 + 订阅聚合**”的模式。

---

## 2. 当前现状判断

### 2.1 已经具备的能力

#### REST 侧

- Smart Cache 已具备 **同键 single-flight**，可以避免同一 `cacheKey` 并发时重复回源
- `fetchBatch()` 已具备 **批次并发控制**
- 强时效 / 弱时效接口已具备不同 TTL 策略
- 本地已有 **Redis 缓存层**，可以承接大量重复命中请求

#### WebSocket 侧

- `StreamConnectionManagerService` 已按 `provider:capability` 维度复用上游连接
- `StreamClientStateManager` 已维护：
  - `client -> symbols`
  - `symbol -> clients`
- Infoway / LongPort 的上游 stream context 已具备“当前连接内的已订阅 symbol 去重”能力

### 2.2 仍然存在的问题

#### REST 侧问题

- 当前没有真正的 **全局上游请求调度器**
- 不同 `cacheKey` 的请求仍可能并发回源
- `fetchBatch()` 的并发限制不等于“按秒限速”
- 当前的 capability `requestsPerSecond` 多数仍是 10，只是元数据，不是强约束执行器
- Redis 已经承担“重复命中返回”，但**还没有承担“相邻请求聚合与统一发车”**

#### WebSocket 侧问题

- 当前“订阅变更监听器”只记录日志，没有把 `symbol -> clients` 聚合关系真正用于上游订阅决策
- 客户端退订时，会直接尝试对上游执行 `unsubscribe`，没有先判断该 symbol 是否仍被其他客户端需要
- JvQuant 当前 `subscribe()` 没有先排除已经订阅的 code，存在重复上游订阅风险

---

## 3. 目标

### 3.1 核心目标

1. **所有 REST 上游请求统一排队**
2. **严格控制上游速率不超过 2 req/s**
3. **尽量把多个项目内请求合并成更少的上游请求**
4. **优先让 Redis/Smart Cache 命中返回，只让真正缓存未命中的请求进入上游请求调度器**
5. **所有 WebSocket 客户端共享上游连接与上游 symbol 订阅**
6. **任何 symbol 只在 `0 -> 1` 时向上游订阅，只在 `1 -> 0` 时向上游退订**

### 3.2 非目标

- 不重写现有 Smart Cache / Stream Cache 架构
- 不引入复杂 MQ 体系作为第一阶段前提
- 不在第一阶段处理所有 provider 的高级自适应策略

---

## 4. 总体设计原则

### 4.1 KISS

- 在现有架构上新增两层协调器，不推翻现有 caching / fetching / stream 结构
- 先解决“打爆上游”的核心风险，再考虑高级优化

### 4.2 YAGNI

- 第一阶段只实现：
  - 上游 REST 总量闸门
  - 上游 WS symbol 聚合
  - 429 冷却与兜底
- 不提前设计复杂优先级调度平台

### 4.3 DRY

- 所有上游 REST 请求统一从一个调度器出去
- 所有上游 WS 订阅统一从一个聚合器出去
- 所有 Redis 命中仍统一走现有缓存模块，不新开第二套重复缓存语义

### 4.4 SOLID

- `DataFetcherService` 继续负责“取数编排”，不直接承担限流队列职责
- `StreamReceiverService` 继续负责入口编排，不直接承担 symbol 引用计数职责
- 新增协调服务负责单一职责的调度和聚合

## 5. REST 方案：上游请求总量闸门

## 5.1 新增组件

建议新增：

- `src/core/03-fetching/data-fetcher/services/upstream-request-scheduler.service.ts`
- `src/core/03-fetching/data-fetcher/interfaces/upstream-request-task.interface.ts`
- `src/core/03-fetching/data-fetcher/constants/upstream-scheduler.constants.ts`

## 5.2 组件职责

`UpstreamRequestSchedulerService` 只负责：

- 按 `provider:capability` 建立独立队列
- 控制请求发往上游的最小时间间隔
- 控制队列长度、等待时间、优先级
- 提供 429 冷却窗口
- 提供“短时间聚合”能力
- 输出调度指标

它**不负责**：

- Redis / Smart Cache 命中判断
- 数据转换
- DTO 组装
- 业务异常翻译

## 5.3 队列粒度

第一阶段建议：

- 默认 key：`provider:capability`

可选增强：

- 对 `get-stock-quote` / `get-stock-history` 这类高度依赖市场特征的能力，可扩展为：`provider:capability:market`

原因：

- 过细会导致队列碎片化
- 过粗会导致不同能力互相阻塞
- `provider:capability` 是当前最平衡的第一阶段方案

## 5.4 限速策略

### 推荐配置（当前代码实际落地）

- `UPSTREAM_SCHEDULER_DEFAULT_RPS = 2`
- `minIntervalMs = floor(1000 / DEFAULT_RPS) = 500`（内部推导，不单独暴露环境变量）
- 单 `queueKey(provider:capability)` 当前为串行发车，等价于 `maxConcurrentExecutions = 1`（内部行为，不单独暴露环境变量）

说明：

- 如果你要“绝对保守”，可以把 `UPSTREAM_SCHEDULER_DEFAULT_RPS` 配成 `1`，等价于 `1000ms` 一次
- 但基于上游提供的 2 次/秒额度，更合理的是 **500ms 发一次**
- “并发数 = 1” 只是补充约束，核心仍然是 **时间速率控制**

## 5.5 调度顺序

默认采用：

- **FIFO**

建议增加简单优先级：

- `P0`：`receiver/data` 实时请求
- `P1`：`query/execute` 查询请求
- `P2`：后台刷新 / 预热 / 补偿任务

实现方式：

- 同一队列内按 `priority -> enqueueTime` 排序
- 第一阶段不需要复杂优先级抢占

## 5.6 接入位置

建议在 `DataFetcherService.fetchRawData()` 内部，对真正调用上游 capability 的那一步进行包装。

目标形态：

```ts
return this.upstreamRequestScheduler.schedule({
  provider,
  capability,
  market,
  priority,
  task: () => this.executeCapability(...),
});
```

也就是说：

- Smart Cache / Basic Cache / Redis 仍然先判断是否命中
- 只有确定需要回源时，才进入 scheduler
- scheduler 是“缓存之后、上游之前”的总闸门

## 5.7 必须叠加的 3 个增强

### 增强 A：请求合并（批量化）

对支持批量 symbols 的能力，特别是：

- `get-stock-quote`
- `get-stock-basic-info`
- `get-stock-history`（仅当未来放开单标的限制时）

可增加一个很短的聚合窗口，例如：

- `mergeWindowMs = 30 ~ 80ms`

在该窗口内，把多个小请求合并为一次上游批量请求。

示例：

- 请求 A：`[00700.HK]`
- 请求 B：`[00941.HK]`
- 请求 C：`[AAPL.US]`

如果同 provider / capability / market 且参数兼容，可以合并成一次批量请求。

### 增强 B：429 冷却窗口

一旦某个 `provider:capability` 收到 429：

- 进入冷却期，例如 `3s ~ 10s`
- 冷却期间：
  - 不再立即打上游
  - 优先返回 stale cache
  - 或让低优先级任务等待 / 超时失败

### 增强 C：stale-if-error / stale-while-revalidate

建议为下列接口增加更明确的兜底策略：

- `get-stock-basic-info`
- `get-market-status`
- `get-trading-days`
- `support-list`

策略：

- 过期缓存仍可在上游失败时短时间兜底返回
- 避免因为上游额度耗尽而把请求全部打成 500

## 5.8 Redis 与调度器的职责边界

这是本方案最重要的补充之一。

### Redis 负责什么

项目本地已有 Redis 缓存层，应该继续优先承担：

- 重复命中直接返回
- 热门 key 的短 TTL 复用
- stale fallback 数据来源
- Query / Receiver 两条链路的已有缓存策略承接

也就是说：

- **能从 Redis/Smart Cache 命中的请求，不应该再进入上游调度器**

### 调度器负责什么

调度器只负责：

- 处理 **缓存未命中后** 的回源请求
- 把多个缓存未命中请求做统一排序和限速
- 对可兼容请求做批量聚合
- 保证真实上游出流速率受控

### 正确执行顺序

建议顺序修正为两类链路：

#### A. 已接入 Smart Cache / Redis 的入口链路

1. 查 Smart Cache / Basic Cache / Redis
2. 命中则直接返回
3. 缓存未命中则进入 single-flight
4. single-flight 未覆盖的请求进入 scheduler
5. scheduler 在短窗口内做 merge
6. merge 后再按 2 req/s 向上游发车

#### B. 直接调用 `DataFetcherService.fetchRawData()` 的链路

例如：

- `chart-intraday` 历史基线拉取
- `support-list` provider 拉取

这类链路未必先经过 Smart Cache 主链，因此应由 scheduler 作为统一兜底：

1. 进入 `fetchRawData()`
2. 按 capability allowlist 判断是否启用 scheduler
3. 启用时进入 single-flight / scheduler / merge
4. 再发往真实上游

因此，不是“用 scheduler 替代缓存”，而是：

- **缓存负责减少重复请求**
- **上游请求调度器（scheduler）负责控制剩余缓存未命中请求以及直连 fetchRawData 链路的发车节奏**

## 5.9 Infoway 专项：哪些能力适合合并

这一节用于指导第一阶段真正落地的批量聚合规则。

### A. `get-stock-quote`

- 上游端点：`GET /stock/batch_trade/{codes}`
- 可合并性：**强可合并**
- 原因：上游天然支持批量 code；项目当前实现本身也会把多个 symbol 拼成一次请求
- 推荐 mergeKey：
  - `provider=infoway + capability=get-stock-quote`
- 推荐 mergeWindow：
  - `30ms ~ 50ms`
- 推荐 maxSymbolsPerBatch：
  - 先按 `INFOWAY_SYMBOL_LIMIT.REST` 控制
  - 再叠加 URL 长度保护
- 备注：
  - 当前 `InfowayContextService` 已有 path 长度切分能力，因此它很适合作为第一批接入 scheduler merge 的能力

### B. `get-stock-basic-info`

- 上游端点：`GET /common/basic/symbols/info?type=...&symbols=...`
- 可合并性：**强可合并，但不建议在调度器层重复计算 `type`**
- 推荐 mergeKey：
  - 第一阶段：`provider=infoway + capability=get-stock-basic-info + marketHint`
  - 如无明确 `marketHint`：允许退化为 `provider=infoway + capability=get-stock-basic-info`
- 推荐 mergeWindow：
  - `50ms ~ 80ms`
- 推荐 maxSymbolsPerBatch：
  - `500`
- 备注：
  - `type` 推导规则已经存在于 `InfowayContextService.getStockBasicInfo()` 中
  - 调度器层应尽量只负责聚合与发车，不要再复制一套 `type` 推导逻辑
  - 真正按 `type` 分桶仍应复用现有 provider 内部实现

### C. `get-market-status`

- 上游端点：`GET /common/basic/markets`
- 可合并性：**最适合合并/复用**
- 推荐策略：
  - 直接把一次全量返回缓存到 Redis
  - 所有请求从这份全量数据里本地过滤
- 推荐 mergeKey：
  - `provider=infoway + capability=get-market-status`
- 推荐 mergeWindow：
  - 可以不做短窗 merge，直接依赖 Redis 全量缓存即可
- 备注：
  - 这是最值得优先做“全量拉一次、本地复用很多次”的能力

### D. `get-trading-days`

- 上游端点：`GET /common/basic/markets/trading_days?market=...&beginDay=...&endDay=...`
- 可合并性：**条件可合并**
- 推荐 mergeKey：
  - `provider=infoway + capability=get-trading-days + market + beginDay + endDay`
- 第一阶段建议：
  - 只合并参数完全相同的请求
- 第二阶段可选增强：
  - 同 `market` 下，把区间做并集后一次请求，再拆给各调用方
- 备注：
  - 区间并集合并是高价值优化，但不建议在第一阶段就上

### E. `get-support-list`

- 上游端点：`GET /common/basic/symbols?type=...&symbols=...`
- 可合并性：**强可合并**
- 当前项目实际使用方式：
  - `support-list` 主链主要是按 `type` 拉全量
- 推荐 mergeKey：
  - `provider=infoway + capability=get-support-list + type`
- 推荐策略：
  - 优先由后台同步 / 长 TTL 缓存承接
  - 前台查询尽量不直接走实时上游
- 备注：
  - 这是典型“同步优先、请求复用优先”的能力，不应该占用实时额度

### F. `get-stock-history`

- 上游端点：`POST /stock/v2/batch_kline`
- 上游能力角度：**支持批量**
- 当前项目实现角度：**暂不适合合并**
- 原因：
  - 你们当前 capability 已明确限制为 `HISTORY_SINGLE = 1`
  - `chart-intraday` 也按单 symbol baseline 拉取
- 当前建议：
  - 先只做同参数 single-flight
  - 不做多 symbol 批量 merge
- 如果未来放开：
  - mergeKey 至少要包括
    - `provider`
    - `capability`
    - `market`
    - `klineType`
    - `klineNum`
    - `timestamp`

## 5.10 Infoway 第一阶段落地优先级

建议优先顺序：

1. `get-market-status`
2. `get-stock-quote`
3. `get-stock-basic-info`
4. `get-support-list`
5. `get-trading-days`
6. `get-stock-history`

原因：

- `get-market-status` 最容易直接靠 Redis 全量缓存见效
- `get-stock-quote` 请求量最高、合并收益最大
- `get-stock-basic-info` 天然适合按 `type` 聚合
- `support-list` 更适合被同步链路吃掉，不应频繁打上游
- `get-trading-days` 合并价值存在，但没有 quote/basic-info 那么高频
- `get-stock-history` 当前受项目实现限制，先别动批量合并

## 6. WebSocket 方案：上游 symbol 订阅聚合器

## 6.1 新增组件

建议新增：

- `src/core/03-fetching/stream-data-fetcher/services/upstream-symbol-subscription-coordinator.service.ts`
- `src/core/03-fetching/stream-data-fetcher/interfaces/upstream-symbol-ref.interface.ts`
- `src/core/03-fetching/stream-data-fetcher/constants/upstream-subscription.constants.ts`

## 6.2 组件职责

该组件不应再维护第二份完整的客户端订阅状态，而应：

- 复用 `StreamClientStateManager` 作为 `client -> symbols` 与 `symbol -> clients` 的事实来源
- 只额外维护：
  - `provider:capability:symbol -> pendingUnsubscribeTimer`
  - 可选的 `provider:capability:symbol -> lastUpstreamActionAt`

并提供两个核心动作：

- `acquire(symbols, clientId, provider, capability)`
- `release(symbols, clientId, provider, capability)`

其中：

- 是否为 `0 -> 1`，应通过 `StreamClientStateManager` 现有状态判断
- 是否为 `1 -> 0`，也应通过 `StreamClientStateManager` 现有状态判断
- 协调器只负责生成“需要对上游执行的增量动作”，不要重复保存一整份 `clientSet` 和 `refCount`

## 6.3 正确的订阅规则

### 订阅

只有当某个 symbol 的引用计数发生：

- `0 -> 1`

才允许真正调用上游：

- `streamDataFetcher.subscribeToSymbols(...)`

如果是：

- `1 -> 2`
- `2 -> 3`

则只更新本地聚合状态，不再向上游重复发订阅。

### 退订

只有当某个 symbol 的引用计数发生：

- `1 -> 0`

才允许真正调用上游：

- `streamDataFetcher.unsubscribeFromSymbols(...)`

如果是：

- `3 -> 2`
- `2 -> 1`

则绝对不能向上游发退订。

## 6.4 延迟退订（强烈建议）

对 `1 -> 0` 的 symbol，不建议立即退订上游。

建议新增：

- `unsubscribeGraceMs = 15000 ~ 30000`（延迟退订窗口，grace period）

行为：

1. 最后一个客户端离开时，不立刻上游退订
2. 标记为待退订
3. 若 grace period 内有新客户端重新订阅，则取消退订
4. 超时后仍然无人订阅，再发上游退订

这样可以显著减少：

- 页面切换
- 短断线重连
- 移动端网络抖动

带来的重复订阅/退订震荡。

## 6.5 接入位置

当前 `StreamReceiverService` 中：

- `subscribeStream()`
- `unsubscribeStream()`

都应改成先调用 `UpstreamSymbolSubscriptionCoordinatorService`，而不是直接碰上游订阅。

目标形态：

```ts
const upstreamSymbolsToSubscribe = coordinator.acquire(...);
if (upstreamSymbolsToSubscribe.length > 0) {
  await streamDataFetcher.subscribeToSymbols(connection, upstreamSymbolsToSubscribe);
}
```

```ts
const upstreamSymbolsToUnsubscribe = coordinator.release(...);
if (upstreamSymbolsToUnsubscribe.length > 0) {
  await streamDataFetcher.unsubscribeFromSymbols(connection, upstreamSymbolsToUnsubscribe);
}
```

## 6.6 与现有 `StreamClientStateManager` 的关系

当前 `StreamClientStateManager` 已有：

- `symbolToClients`
- `getAllRequiredSymbols()`

建议：

- `StreamClientStateManager` 保留为“客户端视角状态管理器”与事实状态源
- 新增的 `UpstreamSymbolSubscriptionCoordinatorService` 作为“上游动作协调器”

不要把两种职责强行混在一个类里，也不要在协调器里复制保存一整套客户端订阅状态，否则后续维护会更复杂。

## 7. 需要顺手修正的现有问题

## 7.1 JvQuant 重复订阅去重

当前 JvQuant `subscribe()` 在发 `add` 前，没有先过滤市场内已订阅 code。

建议修正为：

- 对每个 market，先从 `subscribedCodesByMarket` 中排除已存在 code
- 只有真正新增的 code 才向上游发 `add`

## 7.2 退订误伤问题

当前客户端退订时，会直接尝试对上游执行 `unsubscribe`。

这会导致：

- A 和 B 都订阅 `00700.HK`
- A 退订时，B 可能被误伤

该问题应由新的 symbol 聚合器彻底接管并修复。

## 7.3 capability 元数据与真实额度不一致

当前不少 capability 仍声明：

- `requestsPerSecond: 10`

但你的真实上游额度是：

- `2 req/s`

建议：

- 第一阶段先在 scheduler 层配置真实值，不依赖 capability 元数据
- 第二阶段再统一收敛 metadata，避免误导排查与监控

## 8. 缓存策略调整建议

项目已有缓存较多，不建议再引入全新缓存体系，建议改为“**对现有缓存策略做更保守的上游保护**”。

## 8.1 核心原则：先吃本地缓存，再谈上游调度

这一点需要明确写死：

- Redis / Smart Cache / Basic Cache 已经是第一道防线
- 调度器不是为了替代本地缓存，而是为了处理 **缓存 miss 的剩余流量**
- 因此“是否可以合并”的前提不是全量入口请求，而是：
  - **一段时间内所有缓存 miss 的、且参数兼容的请求**

换句话说：

- 能从 Redis 命中的，直接返回
- 命不中的，再进入 merge + queue + upstream

## 8.2 不同能力的建议策略

### `get-stock-quote`

- 强时效接口仍可保留短 TTL
- 但要配合：
  - single-flight
  - scheduler
  - 批量聚合
  - stale-if-error

### `get-stock-basic-info`

- 明显适合更长 TTL
- 建议默认分钟级甚至小时级
- 可允许 stale fallback

### `get-market-status`

- 强烈建议分钟级缓存
- 开盘前后可稍微缩短 TTL
- 失败时优先返回最近缓存
- 最佳策略是 Redis 中保存一份“全量市场表”

### `get-trading-days`

- 典型低频数据
- 适合长 TTL，甚至日级缓存
- 不应占用高频上游额度

### `get-stock-history`

- 对相同参数应优先复用缓存
- 若是日内分时，可考虑按时间粒度拆 key
- 避免对同一 symbol / timestamp 重复回源

### `support-list`

- 继续保持版本化 + 长周期同步
- 避免进入实时请求通道
- 优先由同步结果和 Redis 承接

## 8.3 推荐新增策略

建议在 Smart Cache 基础上增加两个明确语义：

- `allowStaleOnUpstreamRateLimit`
- `allowStaleOnUpstreamError`

这样可以在调度器里统一决策，而不是在各个业务入口散落 if/else。

## 9. 多实例部署注意事项

如果系统只有 **单实例部署**：

- 第一阶段可以直接使用内存队列与内存 refCount

如果系统未来会 **多实例部署**：

- REST 调度器要迁移为 Redis 分布式限速/队列
- WS symbol 聚合要迁移为 Redis 分布式 refCount 或单独的流聚合节点

否则会出现：

- A 实例按 2 req/s 发
- B 实例也按 2 req/s 发
- 上游实际收到 4 req/s

因此：

- 单实例可以先落地内存方案
- 但设计时应保留存储抽象，方便后续迁到 Redis

## 15. Infoway 调度器数据结构设计

这一节把上面的策略收敛成可以直接编码的数据结构。

### 15.1 核心对象

建议调度器内部至少维护 4 类状态。

注意：这里描述的是 REST 调度器的数据结构，不包含 WS 客户端订阅真相表；WS 侧仍应复用 `StreamClientStateManager`。

```ts
interface UpstreamScheduleTask {
  taskId: string;
  provider: string;
  capability: string;
  mergeKey: string;
  priority: number;
  enqueueAt: number;
  symbols: string[];
  optionsSignature: string;
  options: Record<string, unknown>;
  execute: () => Promise<unknown>;
  resolve: (value: unknown) => void;
  reject: (error: unknown) => void;
}

interface UpstreamMergeBucket {
  mergeKey: string;
  provider: string;
  capability: string;
  createdAt: number;
  flushAt: number;
  symbols: Set<string>;
  tasks: UpstreamScheduleTask[];
}

interface UpstreamQueueState {
  queueKey: string;
  pending: UpstreamScheduleTask[];
  active: boolean;
  lastDispatchAt: number;
  cooldownUntil: number;
}

interface UpstreamRateLimitState {
  requestsPerSecond: number;
  minIntervalMs: number;
  maxQueueSize: number;
  cooldownMsOn429: number;
  // 说明：当前实现未单独保留 maxConcurrentExecutions 字段，
  // 而是通过单 queue 串行发车的行为隐式固定为 1
}
```

### 15.2 状态含义

- `pending`：已经进入队列、但尚未发车的任务
- `mergeBucket`：处于短聚合窗口中的可合并任务组
- `lastDispatchAt`：上一次真正向上游发请求的时间
- `cooldownUntil`：429 后的冷却截止时间

### 15.3 推荐最小实现

第一阶段不需要做复杂优先级堆结构，直接：

- `Map<string, UpstreamQueueState>`
- `Map<string, UpstreamMergeBucket>`

就够了。

其中：

- `queueKey` 负责“限速/发车”
- `mergeKey` 负责“聚合同类请求”

二者不要混成一个概念。

## 16. mergeKey 生成规则

这一节用于避免后续实现时“哪些请求能合并”判断失控。

## 16.1 总原则

只有同时满足下面 3 个条件，才允许合并：

1. **同一个 provider**
2. **同一个 capability**
3. **同一个上游参数签名**

这里最关键的是：

- `mergeKey` 必须只包含**真正影响上游响应结果**的字段
- 不能把无关字段放进去，否则会过度碎片化
- 也不能漏关键字段，否则会把不兼容请求错误合并

## 16.2 推荐拆分

### A. QueueKey

用于限速发车：

```ts
queueKey = `${provider}:${capability}`;
```

必要时可以扩展：

```ts
queueKey = `${provider}:${capability}:${market}`;
```

但第一阶段建议先保持简单。

### B. MergeKey

用于短窗聚合：

```ts
mergeKey = stableStringify({
  provider,
  capability,
  market,
  signature,
});
```

其中 `signature` 是 capability 专属参数签名。

## 16.3 Infoway 各能力的 signature 建议

### `get-stock-quote`

```ts
signature = {
  assetClass: 'stock',
};
```

说明：

- 当前项目实现都走股票 `batch_trade`
- 不需要把 symbol 放进 signature
- symbol 应进入 bucket 的 `symbols` 集合，不应参与 mergeKey 本身

### `get-stock-basic-info`

```ts
signature = {
  marketHint: market ?? "",
};
```

说明：

- 第一阶段不建议在调度器层提前计算 `type`
- `type` 推导和分桶应继续复用 `InfowayContextService.getStockBasicInfo()`
- 调度器只保留足够粗的兼容性签名，避免重复造轮子

### `get-market-status`

```ts
signature = {
  mode: 'full-market-table',
};
```

说明：

- 这个能力建议统一拉全量
- 不建议把请求中的 `market` 作为 mergeKey 的拆分条件
- `market` 只影响本地下游过滤，不影响上游发车目标

### `get-trading-days`

```ts
signature = {
  market,
  beginDay,
  endDay,
};
```

说明：

- 第一阶段只合并完全相同参数
- 后续如果做区间并集优化，再单独升级这部分逻辑

### `get-support-list`

```ts
signature = {
  type,
  mode: symbols?.length ? 'partial' : 'full',
};
```

说明：

- 当前主链更接近 `full by type`
- 经本轮复审，`support-list` 当前实现已满足项目目标，本轮不纳入 Phase 3 改造范围
- 如果将来真的支持 `symbols` 子集过滤，`partial` 与 `full` 不能混合并

### `get-stock-history`

```ts
signature = {
  market,
  klineType,
  klineNum,
  timestamp,
  mode: 'single-symbol',
};
```

说明：

- 当前项目实现已限定单标的
- 所以第一阶段即使进入 scheduler，也只做同参数单飞，不做多 symbol merge

## 16.4 stable stringify 要求

实现 `signature` 序列化时，必须保证：

- 字段顺序稳定
- 空值策略一致
- 数字/字符串类型归一

否则 mergeKey 会抖动，导致本可合并的请求被拆散。

建议单独提供一个：

- `buildUpstreamMergeKey()`
- `buildUpstreamQueueKey()`
- `buildInfowayCapabilitySignature()`

放在：

- `src/core/03-fetching/data-fetcher/utils/`
