## 1. 文档定位

本文档用于说明当前项目中已经落地的两类协调能力：

1. REST 上游请求调度器
2. WebSocket 上游订阅聚合协调器

本文档覆盖：

- 调度器为什么存在
- 当前实现落在什么位置
- 核心数据结构与运行机制
- Infoway crypto 两轮 `429` 修复后的最终口径
- 缓存、调度、订阅聚合之间的职责边界
- 当前测试与回归方式

本文档不覆盖：

- 各 provider 的字段映射细节
- 对外 API DTO 协议说明
- 完整缓存体系设计文档
- `support-list` 的版本化同步细节

本文档以当前代码事实为准，不把早期方案文档中的“建议项”误写成“当前现状”。

## 2. 背景与问题演进

项目原本已经具备较完整的缓存体系，包括 `Smart Cache`、`Basic Cache`、`Stream Cache` 等能力，但这并不能完全解决“上游额度很紧”的问题。

在缺少统一调度之前，系统仍存在两个核心风险：

1. REST 请求即使命中缓存比例很高，只要出现不同 key 的并发 miss，仍可能同时回源并瞬时冲击上游。
2. WebSocket 虽然已经做了连接复用，但如果没有基于 symbol 的真实引用计数聚合，多个客户端仍可能触发重复的上游订阅与误退订。

围绕这个问题，方案经历了三次收敛：

1. 总方案阶段  
   通过“上游请求削峰与流订阅聚合方案”明确提出：缓存之后、上游之前需要一层总闸门；连接复用之外还需要一层上游 symbol 订阅聚合。
2. Crypto 第一次修复  
   发现 `provider:capability` 粒度不足以约束 Infoway crypto REST，因为其真实额度更接近 provider 级共享池，因此新增 `dispatchScopeKey`，把 `quote / basic-info / history` 收敛到 `infoway:crypto-rest`。
3. Crypto 第二次修复  
   共享调度域生效后，`scheduler-only` 首轮仍偶发单次 `429`。最终通过为 `infoway:crypto-rest` 单独覆盖更保守的最小发车间隔，解决贴边发车问题。

当前实现的真实口径是：

- 项目已经不是“只有缓存，没有总闸门”
- 项目也不再是“crypto 没接入调度器”
- 当前状态是“缓存 + REST 调度器 + WS 订阅聚合 + crypto 共享调度域间隔覆盖”

### 3.2 REST 主链路

当前 REST 主链路可以抽象为：

```text
HTTP Entry
  -> Cache / Single-Flight
  -> DataFetcherService
  -> UpstreamRequestSchedulerService
  -> Provider Capability
  -> Transformer / Response
```

需要强调：

- scheduler 不替代 cache
- scheduler 只处理 cache miss 之后的回源请求，或直接进入 `fetchRawData()` 的 allowlist 链路
- provider capability 仍负责真正的上游调用与 provider 侧拆批细节

### 3.3 WebSocket 主链路

当前 WS 主链路可以抽象为：

```text
WebSocket Entry
  -> StreamReceiverService
  -> ConnectionManager
  -> StreamClientStateManager
  -> UpstreamSymbolSubscriptionCoordinatorService
  -> StreamDataFetcher
  -> Provider Stream Context
```

需要强调：

- 连接复用和 symbol 聚合不是一回事
- `StreamClientStateManager` 负责事实状态
- `UpstreamSymbolSubscriptionCoordinatorService` 只负责把事实状态转换为“是否真的要对上游执行 subscribe / unsubscribe”

## 5. 职责边界

### 5.1 Cache 负责什么

`Smart Cache` / `Basic Cache` / 既有 single-flight 负责：

- 优先命中已有缓存
- 合并同 key 并发请求
- 为低实时性能力提供 stale 数据来源

这些能力的目标是减少“本不该发生的回源”。

### 5.2 Scheduler 负责什么

`UpstreamRequestSchedulerService` 负责：

- 在 REST 回源前做统一排队
- 在短窗口内合并兼容请求
- 对真实发车节奏做最小间隔控制
- 在收到 `429` 后进入冷却窗口
- 对支持按 symbol 回填的能力做结果拆分

它不负责：

- 判断缓存是否命中
- 存储响应缓存
- 组装 DTO
- 翻译业务字段
- 替代 provider 内部拆批逻辑

### 5.3 DataFetcherService 负责什么

`DataFetcherService` 负责回源编排，而不是维护调度状态。它的职责包括：

- 校验 provider/capability 是否存在
- 规范化 `apiType`
- 决定是否进入 scheduler
- 为不同能力注入正确的 `mergeMode`
- 为 Infoway quote/crypto quote 注入正确的 `symbolExtractor`
- 在上游失败时执行 stale fallback 与错误翻译

### 5.4 WS 协调器负责什么

`UpstreamSymbolSubscriptionCoordinatorService` 只负责两件事：

1. 判断某批 symbol 是否真的需要向上游发订阅动作
2. 判断某批 symbol 是否真的需要向上游发退订动作

它不维护第二份完整的客户端订阅真相表。真实状态仍由 `StreamClientStateManager` 提供。

### 5.5 不应该由谁负责

以下职责不能混写：

- 不能让 scheduler 负责缓存命中判定
- 不能让 `StreamReceiverService` 自己维护另一套 symbol refCount 真相
- 不能把 crypto 共享调度域理解成“共享 merge bucket”
- 不能把 `support-list` 重新拉回实时调度主链

## 6. REST 调度器核心模型

### 6.1 术语定义

`allowlist`

- 控制哪些 `provider:capability` 允许进入 scheduler
- 当前运行时优先读取 `UPSTREAM_SCHEDULER_ALLOWLIST`
- 若环境变量为空，则回退到代码常量默认集合

`queueKey`

- 语义标识，格式为 `provider:capability`
- 用于日志、诊断、能力边界说明
- 不一定等于真实排队的队列键

`dispatchScopeKey`

- 真实排队、发车节奏与 `429` 冷却的作用域
- 未命中映射时，等于 `queueKey`
- 命中共享调度域映射时，多个 capability 共用同一个发车域

`mergeKey`

- 用于判断请求能否在短窗口内合并
- 基于 `provider + capability + signature` 生成
- 使用 `stableStringify()` 保证字段顺序稳定

`mergeMode`

- 当前支持两种：
  - `merge_by_request_signature`
  - `single_symbol_only`

`lastDispatchAt`

- 某个 `dispatchScopeKey` 最近一次真正发车时间

`cooldownUntil`

- 某个 `dispatchScopeKey` 最近一次 `429` 后的冷却截止时间

### 6.2 为什么 `queueKey` 和 `dispatchScopeKey` 必须分离

这是 crypto 修复的关键。

如果只保留 `queueKey=provider:capability`：

- `get-crypto-quote`
- `get-crypto-basic-info`
- `get-crypto-history`

会各自独立排队、各自独立计算发车间隔、各自独立计算 `429` 冷却。对于真实共享额度池来说，这种粒度过细，无法拦住跨 capability 的混合并发。

引入 `dispatchScopeKey` 之后：

- `queueKey` 仍保留 capability 语义
- `dispatchScopeKey` 才是真正的共享发车域

这样既能共享节奏控制，又不会把 mergeKey 和结果分流语义揉乱。

### 6.3 调度器内部状态

当前调度器内部核心维护以下几类状态：

- `queues: Map<dispatchScopeKey, UpstreamQueueState>`
- `mergeBuckets: Map<mergeKey, UpstreamMergeBucket>`
- `activeEntries: Map<mergeKey, UpstreamDispatchEntry>`
- `stats`

其中：

- `queues` 决定真实的排队、串行发车与冷却
- `mergeBuckets` 决定短时间内哪些请求先聚合再入队
- `activeEntries` 允许已经在飞、且 symbols 已覆盖的请求继续复用当前发车

### 6.4 key 构造规则

当前实现通过以下工具函数构造 key：

- `buildUpstreamQueueKey(provider, capability)`
- `buildUpstreamMergeKey(provider, capability, signature)`
- `stableStringify(value)`

`stableStringify()` 会对对象 key 做稳定排序，并忽略 `undefined` 字段，避免本可合并的请求因为对象序列化顺序不同而产生不同的 mergeKey。

## 7. REST 调度执行链路

### 7.1 DataFetcher 接入判定

`DataFetcherService.fetchRawData()` 在真正执行 capability 之前，会先进入 `executeWithOptionalScheduling()`。

只有同时满足以下条件，才会进入 scheduler：

1. 已注入 `UpstreamRequestSchedulerService`
2. `apiType` 被规范化为 `rest`
3. `shouldSchedule(provider, capability, apiType)` 返回 `true`
4. 对应 `provider:capability` 命中 allowlist

否则直接执行原始 capability 调用。

### 7.2 mergeMode 注入

当前 `DataFetcherService` 会在进入 scheduler 前，为能力注入不同 mergeMode：

- `infoway:get-stock-history` -> `single_symbol_only`
- `infoway:get-crypto-history` -> `single_symbol_only`
- 其它当前 allowlist 能力 -> `merge_by_request_signature`

这意味着：

- history 类能力在调度器层面就会被强制保护为单标模式
- 即使未来上游支持 batch，当前项目实现也不会在 scheduler 中把多个 history symbol 合并到一次发车

### 7.3 symbolExtractor 注入

当前只有 Infoway quote 类能力注入特殊 symbol 提取器：

- `infoway:get-stock-quote`
- `infoway:get-crypto-quote`

原因是 Infoway quote 原始行里常用 `s` 字段而不是标准 `symbol` 字段。  
调度器需要能够在合并返回后，按每个下游 task 的 symbol 集合正确回填结果。

### 7.4 attachTask 行为

请求进入 scheduler 后，调度器会按以下顺序尝试复用：

1. 先尝试并入现有 `mergeBucket`
2. 再尝试并入同 `dispatchScopeKey` 下尚未发车的 pending entry
3. 再尝试加入同 `mergeKey` 的 active entry
4. 如果都不能复用，则新建 `mergeBucket`

这一点非常重要，因为它决定了系统不仅能在短窗内 merge，还能复用已经入队但尚未发车、甚至已经在飞且 symbol 集合足够覆盖的请求。

### 7.5 merge bucket 刷新

当前 merge window 规则如下：

- `quote` / `crypto-quote`: `50ms`
- `stock-basic-info` / `crypto-basic-info`: `80ms`
- 其它当前能力：`0ms`

当 bucket 到期后，会被 flush 为一个 `UpstreamDispatchEntry`，按其 `dispatchScopeKey` 入队。

### 7.6 processQueue 行为

调度器对每个 `dispatchScopeKey` 都是串行发车：

1. 若队列已 active，则不重复启动处理循环
2. 发车前先执行 `waitForDispatchWindow()`
3. 满足窗口后取出队首 entry
4. 设置 `lastDispatchAt`
5. 调用 `dispatchEntry()`
6. 如果遇到 `429`，则更新该 `dispatchScopeKey` 的 `cooldownUntil`

当前发车顺序是严格 FIFO，没有实现 capability 优先级调度。

### 7.7 结果回填

对于以下能力，调度器会在一次合并发车后，再按 symbol 拆回各 task：

- `get-stock-quote`
- `get-stock-basic-info`
- `get-crypto-quote`
- `get-crypto-basic-info`

对于不支持按 symbol 回填的能力，调度器会把同一结果直接 resolve 给合并 entry 中的所有 task。

### 7.8 `429` 与异常传播

如果 scheduler 层收到上游 `429`：

1. 当前 `dispatchScopeKey` 会进入冷却窗口
2. entry 内的所有 task 会收到错误
3. 错误会继续回到 `DataFetcherService.fetchRawData()`
4. `DataFetcherService` 会尝试 stale fallback
5. 如果没有可用 stale 数据，则翻译为 `BusinessException`，状态码 `429`

因此，scheduler 只负责节奏控制，不负责吞掉错误。

## 8. 请求合并策略

### 8.1 quote / crypto quote

当前 quote 类能力具备最明确的短窗 merge 收益：

- merge window: `50ms`
- 支持多 symbol 合并发车
- 结果按 symbol 回填
- Infoway quote 使用 `s` 字段进行结果拆分

### 8.2 stock-basic-info / crypto-basic-info

当前 basic-info 类能力也支持短窗 merge：

- merge window: `80ms`
- 仍按 capability 语义生成 mergeKey
- 合并后的结果按 symbol 回填

需要强调：

- crypto basic-info 进入的是共享 crypto 调度域
- 但它只共享发车节奏与冷却
- 它不会和 crypto quote/history 混成同一个 merge bucket

### 8.3 stock-history / crypto-history

当前 history 类能力统一走 `single_symbol_only`：

- 不同 symbol 不会合并发车
- 同 symbol、同参数的请求可以 single-flight 复用
- 调度器在入参阶段和发车阶段都做了单标保护

这保证了：

- 不会因为短窗合并把多个 history symbol 打成一次请求
- 不会破坏当前项目对 history 单标契约的依赖

### 8.4 get-market-status

`get-market-status` 当前会进入 scheduler allowlist，但在调度器层没有 quote/basic-info 那种专门的短窗 merge 特化，merge window 为 `0ms`。

这意味着：

- 它仍受统一排队和发车节奏保护
- 但当前调度器并没有为它额外实现短窗批量聚合

如需进一步优化这类能力，应优先通过缓存和 provider 层复用策略推进，而不是在本文档里误写成“已经在 scheduler 层做了全量市场表聚合”。

### 8.5 get-trading-days

`get-trading-days` 当前的真实状态是：

- 默认不在 scheduler allowlist
- 但在 `DataFetcherService` 中支持 stale fallback
- stale key 由 `market + beginDay + endDay + symbols(sorted)` 构成

因此，它目前是“stale 兜底能力”，而不是“默认 scheduler 能力”。

## 9. 共享调度域与 Crypto 两轮修复

### 9.1 第一次修复解决了什么

第一次 crypto 修复引入了：

`UPSTREAM_SCHEDULER_DISPATCH_SCOPE_MAP`

当前映射为：

- `infoway:get-crypto-quote -> infoway:crypto-rest`
- `infoway:get-crypto-basic-info -> infoway:crypto-rest`
- `infoway:get-crypto-history -> infoway:crypto-rest`

这解决的是“跨 capability 抢同一上游额度池”的问题。

### 9.2 第二次修复解决了什么

第一次修复之后，crypto 的多个 capability 已经共享同一个发车域，但在 `scheduler-only` 路径下，首轮仍偶发单次 `429`。

第二次修复引入了：

`UPSTREAM_SCHEDULER_DISPATCH_SCOPE_MIN_INTERVAL_MS_MAP`

当前映射为：

- `infoway:crypto-rest -> 1300`

这意味着：

- crypto 共享域不再直接沿用全局默认发车间隔
- 即使全局默认 `DEFAULT_RPS` 变化，crypto 共享域仍使用更保守的 `1300ms`

### 9.3 当前默认值的真实关系

当前 scheduler 常量默认值为：

- `UPSTREAM_SCHEDULER_DEFAULT_RPS = 2`
- 非覆盖调度域默认最小发车间隔约为 `500ms`
- `infoway:crypto-rest` 固定覆盖为 `1300ms`

这与历史文档里的“共享域曾继承 `1000ms`”并不矛盾：

- `1000ms` 是当时问题定位环境的全局节奏
- 当前代码已经把默认值回到 `2 req/s`
- 但 crypto 共享域仍然保留更保守的独立覆盖

### 9.4 共享了什么，没有共享什么

crypto 共享调度域当前只共享：

- 发车顺序
- 最小发车间隔
- `429` 冷却窗口

没有共享：

- merge bucket
- mergeKey
- 响应数据
- history 的单标契约
- quote/basic-info 的结果分流逻辑

这就是为什么“共享调度域”不能被误解为“跨 capability merge”。

## 10. 缓存、调度与 stale fallback 的边界

### 10.1 正确顺序

对于标准 REST 入口，应理解为：

```text
Cache Hit -> 直接返回
Cache Miss -> single-flight
single-flight 仍需回源 -> scheduler
scheduler 发车 -> upstream
upstream 失败 -> stale fallback（若能力支持）
```

### 10.2 stale fallback 当前范围

当前 stale fallback 仅对以下能力生效：

- `infoway:get-stock-basic-info`
- `infoway:get-trading-days`

默认 TTL：

- `UPSTREAM_BASIC_INFO_STALE_TTL_SECONDS = 3600`
- `UPSTREAM_TRADING_DAYS_STALE_TTL_SECONDS = 86400`

这意味着：

- `quote`
- `history`
- `crypto quote`
- `crypto basic-info`
- `crypto history`

当前都不走这套 stale fallback。

### 10.3 support-list 的位置

`support-list` 当前应继续被视为：

- 版本化同步型能力
- 长周期刷新型能力
- 非实时回源主链能力

它当前不属于本文档讨论的实时调度主链，不应继续按 quote/basic-info 的策略理解。

## 11. WebSocket 订阅聚合机制

### 11.1 事实状态源

`StreamClientStateManager` 当前已经维护：

- `client -> symbols`
- `symbol -> clients`
- `provider+capability+symbol -> clients`
- `provider -> clients`

其中，`upstreamToClients` 对应的 `provider+capability+symbol -> clients` 映射，是当前上游订阅聚合最关键的事实依据。  
`provider -> clients` 则主要用于 provider 级客户端分布与清理辅助索引，不直接参与 `0 -> 1 / 1 -> 0` 的上游订阅判定。

### 11.2 subscribeStream 当前顺序

`StreamReceiverService.subscribeStream()` 当前顺序是：

1. 解析并标准化 symbol
2. 获取或创建连接
3. 调用 `coordinator.acquire(...)`
4. 仅对协调器返回的 symbol 调用真实上游 `subscribeToSymbols(...)`
5. 上游订阅成功后，才写入 `StreamClientStateManager.addClientSubscription(...)`

这个顺序很重要，因为它避免了“上游订阅失败但本地状态已提前落库”的脏状态问题。

### 11.3 acquire 语义

`acquire()` 的语义是：

- 如果某个 `provider:capability:symbol` 当前引用计数为 `0`
- 那么这次订阅会触发真实上游订阅动作

否则：

- coordinator 不返回上游订阅动作
- 后续由 `StreamReceiverService` 在上游订阅成功后写入本地订阅关系

这就是标准的 `0 -> 1` 语义。

### 11.4 unsubscribeStream 当前顺序

`StreamReceiverService.unsubscribeStream()` 当前顺序是：

1. 先从 `StreamClientStateManager` 移除客户端订阅
2. 再调用 `scheduleRelease(...)`
3. 如果是立即释放，则直接对上游执行 `unsubscribe`
4. 如果进入 grace period，则返回 `scheduled release` 信息，延迟执行真正退订
5. 如果当前根本没有活跃连接，则跳过真实上游退订

### 11.5 release 语义

`scheduleRelease()` 的语义是：

- 只有当某个 `provider:capability:symbol` 的引用计数变成 `0`
- 才允许真正进入 release 判定

如果仍有其它客户端订阅同一 `provider:capability:symbol`：

- 不执行上游退订

这就是标准的 `1 -> 0` 语义。

### 11.6 延迟退订

当前默认宽限期配置为：

- `UPSTREAM_SUBSCRIPTION_UNSUBSCRIBE_GRACE_MS = 15000`

作用：

- 避免页面切换、短断线重连、移动网络抖动造成的订阅/退订震荡
- 在宽限期内重新订阅同一 symbol 时，取消待退订 timer

### 11.7 provider 隔离

当前 coordinator 的 key 是：

`provider:capability:symbol`

因此：

- 同一 symbol 跨 provider 订阅时，仍然各自独立判断是否要发真实上游订阅
- 同一 symbol 跨 provider 释放时，也不会互相阻塞

这也是当前单元测试明确覆盖的行为。

### 13.1 当前关键日志

- 活跃连接不存在时跳过上游退订

### 13.2 当前内部统计

`UpstreamRequestSchedulerService` 当前维护以下内部统计：

- `mergedTasks`
- `dispatchedTasks`
- `cooldownHits`
- `maxQueueDepth`

并通过 `getStats()` 暴露。

需要明确：

- 当前代码里这些统计主要是 service 内部可读状态
- 没有现成的统一 exporter 或完整监控面板
- 当前项目监控模块已裁剪，线上建议通过日志和外部监控系统补足

### 13.3 线上排障建议

如果要排查“为什么仍然出现 `429` 或者为什么请求没有进入调度器”，建议按以下顺序看：

1. 看 `apiType` 是否是 `rest`
2. 看对应 `provider:capability` 是否命中 allowlist
3. 看日志里的 `queueKey` 与 `dispatchScopeKey` 是否符合预期
4. 看该共享域是否进入过 cooldown
5. 对比 `smart-cache + scheduler` 与 `scheduler-only` 的行为差异
6. 确认问题是：
   - 缓存未命中导致回源增多
   - 调度域划分不正确
   - 上游真实额度比预期更紧

## 15. 常见误区

### 15.1 误区：命中了 scheduler 就等于一定发生 merge

不对。  
进入 scheduler 只代表请求会受统一排队和节奏控制。是否发生 merge，还取决于：

- capability 是否支持短窗 merge
- mergeKey 是否兼容
- 请求是否在窗口内相遇

### 15.2 误区：共享调度域等于跨 capability 合并

不对。  
共享调度域只共享节奏与冷却，不共享 merge bucket 和响应数据。

### 15.3 误区：`get-trading-days` 已经默认进入 scheduler

不对。  
当前默认配置下，它是 stale fallback 能力，不是默认 scheduler allowlist 成员。

### 15.4 误区：`support-list` 仍然是本次调度器主改造对象

不对。  
`support-list` 当前已经被收口为版本化同步型能力，不应继续混入实时调度主链说明。

### 15.5 误区：有了 scheduler 之后就不需要 cache

不对。  
cache 减少“不必要的回源”，scheduler 控制“不得不发生的回源”。两者是协作关系，不是替代关系。

## 16. 当前限制与后续边界

当前实现明确存在以下边界：

- 主要是单实例内存态调度与聚合
- 没有分布式共享队列
- 没有跨实例共享 refCount
- 没有动态自适应 RPS
- 没有多级优先级调度
- 共享调度域仍然是显式常量映射，不是自动推导

如果未来进入多实例部署，需要补齐至少两件事：

1. REST 调度改造成可跨实例共享的限流/排队机制
2. WS symbol 聚合改造成可跨实例共享的 refCount 机制

否则多实例会分别按自己的本地节奏发车，整体仍可能突破上游真实额度。
