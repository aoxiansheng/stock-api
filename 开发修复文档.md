### 修复目标
1. 恢复 lint 闸门可执行（从“配置报错”变为“真实代码检查结果”）。
2. 保持最小成本改动，不引入额外兼容分支和过度工程。

### 涉及文件
- `/Users/honor/Documents/code/newstockapi/stock-api/eslint.config.mts`

### 影响面分析
- 直接影响链路：`bun run lint` -> ESLint Flat Config 加载 -> 规则解析失败 -> 检查中断。
- 间接风险：团队可能绕过 lint，导致代码规范/安全规则失效。
- 发布风险：CI 闸门不可用时，合并决策失真。

### 原子开发步骤（函数/规则粒度）
1. 在 `eslint.config.mts` 中删除无效规则键 `@typescript-eslint/no-unused-imports`。

2. 执行 `bun run lint` 可完整运行到代码扫描阶段并返回真实结果。
3. 执行 `bun run test:unit -- "test/unit/providersv2/provider-registry.service.spec.ts"` 通过（回归确认配置改动未影响运行时逻辑）。

### 测试补齐策略

## FIX-002（P0）
### 背景
`test/unit/_verification/critical-group.verification.spec.ts` 的 4 条断言仍在验证历史缺陷行为（标准符号应 reject、token 应残留），与当前实现目标冲突，造成假红并降低测试可信度。

### 修复目标
1. verification 测试转为验证“当前正确行为契约”。
2. 消除反向断言对安全与业务逻辑的误导。

### 涉及文件
- `/Users/honor/Documents/code/newstockapi/stock-api/test/unit/_verification/critical-group.verification.spec.ts`

### 影响面分析
- 业务链路影响：`QueryExecutionEngine` -> `ReceiverService` -> `DataTransformerService.restoreStandardSymbols`。
- 安全链路影响：`sanitizeInfowayUpstreamMessage` 在 Infoway REST/WS 错误日志链路中的脱敏有效性。
- 发布风险：关键验证分组长期假红会导致测试闸门失真。

### 原子开发步骤（函数级断言）
1. 将 `restoreStandardSymbols` 的 3 条参数化断言从 `rejects` 改为 `resolves`，并断言返回 `symbol` 与输入标准符号一致。
2. 将 `sanitizeInfowayUpstreamMessage` 用例断言改为：结果包含 `Authorization=[REDACTED]` 且不包含原 token。
3. 在文件头增加一句契约说明：该文件验证“现状行为”，不验证历史缺陷快照。

### 每步验收条件
1. 执行 `bun run test:unit -- "test/unit/_verification/critical-group.verification.spec.ts"` 全通过。
2. 执行 `bun run test:unit -- "test/unit/core/02-processing/transformer/services/data-transformer.service.restore-symbols.spec.ts"` 通过。
3. 执行 `bun run test:unit -- "test/unit/providersv2/infoway/infoway-error.util.spec.ts"` 通过。
4. 执行组合回归：
   `bun run test:unit -- "test/unit/_verification/critical-group.verification.spec.ts" "test/unit/core/02-processing/transformer/services/data-transformer.service.restore-symbols.spec.ts" "test/unit/providersv2/infoway/infoway-error.util.spec.ts"`
   全通过。

### 测试补齐策略
- 不新增测试文件，仅修正既有断言语义。
- 将 verification 保持为跨模块契约验证，模块细节继续依赖现有专用 spec，避免重复测试逻辑。
