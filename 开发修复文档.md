# stock-api 开发修复文档（基于 Multi-Agents 核实结果）

## 1. 背景
A/B 审查汇总指出当前版本不存在 Critical 级问题，但存在 3 个 Important 与 2 个 Minor 问题，集中在：
- 输入边界治理不足（provider 输出缺少统一 schema 校验）
- 门禁顺序不优（限速晚于深度参数校验）
- symbol 校验路径存在空白值放行
- 配置依赖显式性与配置读取重复问题

这些问题会放大为缓存键污染、广播错投、无效流量成本上升、维护复杂度上升等系统性风险。当前版本不建议直接发布，应先完成修复并做定向+全量回归。

## 2. Multi-Agents 拆分与核实结论

### 2.1 拆分策略
- 子 Agent A：核实 I-1（identity 分支 symbol 校验顺序/空白放行）
- 子 Agent B：核实 I-2（provider 输出入站 schema 校验缺失）
- 子 Agent C：核实 I-3（限速门禁顺序）
- 子 Agent D：核实 M-1/M-2（配置依赖显式化与配置读取重复）

### 2.2 核实汇总
| 编号 | 结论 | 说明 |
|---|---|---|
| C-0 | 成立 | 当前未发现 Critical 项 |
| I-1 | 存在 | identity 路径存在空白 symbol 放行风险（根因为前置空白直接 return + 先标准化后校验路径） |
| I-2 | 存在 | provider 输出进入 cache/room/broadcast 前缺少统一格式合法性校验 |
| I-3 | 存在 | 先参数深校验后限速，存在无效请求路径绕过限速统计 |
| M-1 | 存在 | transformer 模块对配置能力依赖不够显式 |
| M-2 | 存在 | stream receiver 内配置读取逻辑重复 |

## 3. 修复目标

### 3.1 总体目标
- 封堵脏数据进入 cache/room/broadcast 的入口
- 把限速前置为轻量门禁，降低无效请求成本
- 修复 identity symbol 校验顺序问题，避免空白/非法值放行
- 降低配置依赖与配置读取的维护风险

### 3.2 非目标（YAGNI）
- 不重写整条 stream 处理链
- 不引入额外复杂中间件框架
- 不改变既有错误码体系与对外接口协议

### 3.3 优先级
- P0：I-2（污染入口治理）
- P1：I-3（门禁顺序）
- P2：I-1（identity 分支校验）
- P3：M-1/M-2（可维护性改进）

## 5. 影响面分析

### 5.1 功能影响
- 订阅流入口将更严格，部分历史“脏但可过”的数据会被丢弃/拒绝
- 限速拒绝可能更早发生（符合防护目标）
- identity provider 对 symbol 输入约束更严格

### 5.2 性能影响
- 增加轻量 schema 校验开销（单条微小增量）
- 通过前置限速减少无效请求后续处理，总体预期净收益

### 5.3 兼容性影响
- 对外 API 协议与错误码不变
- 内部日志与指标新增丢弃原因维度（仅增强可观测性）

### 5.4 风险与缓解
- 风险：过严校验导致误丢弃
- 缓解：先以单元测试+回归锁定规则，并在日志中输出 dropped reason 便于快速回溯

## 6. 分问题修复计划（原子步骤 + 验收条件）

## 6.1 I-2：Provider 输出缺少统一合法性校验（P0）

### 6.1.1 背景证据
- `stream-receiver.service.ts:165`：`pipelineCacheData` 以 `any` 直接写缓存
- `stream-receiver.service.ts:224`：`pipelineBroadcastData` 以 `any` 直接广播
- `stream-receiver.service.ts:1421`：`toCanonicalSymbol` 仅做 trim+upper，不做格式校验
- 上游批处理链路未形成统一 schema 门禁

### 6.1.2 修复目标
在进入 cache/room/broadcast 前，强制通过单一入口的“校验+标准化”流程，确保只下沉合法 canonical 数据。

### 6.1.3 原子步骤与验收条件
1. 新增统一校验器 `validateAndCanonicalizeStreamPayload(...)`。
- 验收：可输出 `validItems/droppedCount/reasons`，且 reasons 分类稳定。
2. 在批处理下沉前统一接入该校验器（单一入口）。
- 验收：`pipelineCacheData` 和 `pipelineBroadcastData` 入参均为合法 canonical 数据。
3. 在 `pipelineCacheData`/`pipelineBroadcastData` 增加轻量断言 `assertCanonicalItem`（防绕过）。
- 验收：绕过统一入口传入脏数据时被拦截并记录告警。
4. 增加 dropped reason 指标与日志字段。
- 验收：可区分 `invalid_symbol/invalid_price/invalid_timestamp/...`。
5. 清理重复字段判定逻辑，保证 DRY。
- 验收：不存在两套不一致校验规则。

### 6.1.4 测试补齐
- 新增：非法 symbol、`price=NaN`、无效 timestamp、负 volume 被丢弃且不缓存/不广播
- 新增：合法数据正常缓存并广播
- 新增：批处理入口统一校验生效（建议补 `stream-batch-processor` 相关测试）

### 6.1.5 回归验证
- `stream-receiver.service.spec.ts` 全量通过
- stream-receiver 相关单测全量通过
- 验证 room key/cache key/broadcast symbol 一致性不回退

## 6.2 I-3：限速门禁顺序晚于深度参数校验（P1）

### 6.2.1 背景证据
- `stream-receiver.service.ts:642` 先 `validateSubscribeRequest`
- `stream-receiver.service.ts:692` 先做 symbol 深校验
- `stream-receiver.service.ts:700` 才执行限速
- 现有测试已固化“非法请求不触发限速”的旧行为

### 6.2.2 修复目标
先执行轻量限速/配额门禁，再执行深度参数校验，减少无效请求资源消耗。

### 6.2.3 原子步骤与验收条件
1. 将连接限速检查前移到 `subscribeStream` 入口最前。
- 验收：不存在先抛 `DATA_VALIDATION_FAILED` 再限速的路径。
2. 保持既有错误码与日志字段语义不变。
- 验收：限速拒绝仍返回既有业务错误码。
3. 更新旧测试预期（非法请求也先经过限速门禁）。
- 验收：受影响存量测试语义一致迁移。
4. 新增顺序断言测试（限速调用顺序早于深校验）。
- 验收：调用顺序断言稳定通过。
5. 新增“超限短路”测试。
- 验收：超限后不进入深校验链路。

### 6.2.4 测试补齐
- 无效 symbol + 有 clientIp 仍先触发限速
- 超限直接拒绝，不触发深校验
- 限速通过 + 参数非法返回 `DATA_VALIDATION_FAILED`

### 6.2.5 回归验证
- `stream-receiver.service.spec.ts` 通过
- stream-receiver 相关测试通过
- 验证 clientIp 缺失分支行为与既有策略一致

## 6.3 I-1：identity symbol 校验顺序/空白放行（P2）

### 6.3.1 背景证据
- `data-transformer.service.ts:683-686`：空白 symbol 提前 `return item`
- `data-transformer.service.ts:693-695`：identity 路径先 normalize 再 `isStandardIdentitySymbol`
- `provider-symbol-identity.util.ts:97-104`：规则要求原串不得有首尾空白，但被先 normalize 路径稀释

### 6.3.2 修复目标
identity provider 对原始 symbol 先做严格校验（非空、格式合法）后再标准化，禁止空白符号透传。

### 6.3.3 原子步骤与验收条件
1. 在 `restoreStandardSymbols/mapSymbol` 中对 identity provider 前置原始值校验。
- 验收：`symbol: "   "`、`"\t\n"` 触发 `DATA_VALIDATION_FAILED`。
2. non-identity provider 行为保持不变。
- 验收：非 identity 既有逻辑回归通过。
3. 保持异常结构与错误码一致。
- 验收：错误对象字段兼容现有处理链。
4. 补测试并回归 transformer 相关测试。
- 验收：新增用例通过，旧用例无回退。

### 6.3.4 测试补齐
- identity + 空白 symbol 抛错
- identity + 非字符串/缺失 symbol（按约定）抛错
- identity 正常 canonicalization 保持通过

### 6.3.5 回归验证
- `data-transformer.service.restore-symbols.spec.ts` 全绿
- transformer 目录相关单测通过

## 6.4 M-1：配置模块隐式依赖（P3）

### 6.4.1 背景证据
- `data-transformer.module.ts:17` 未显式声明配置模块依赖
- service 注入 `ConfigService`，模块装配依赖全局模块链路
- runtime 配置工厂直接读 `process.env`

### 6.4.2 修复目标
显式声明配置依赖，统一通过 `ConfigService` 读取并校验配置，降低隐式耦合。

### 6.4.3 原子步骤与验收条件
1. `TransformerModule` imports 显式引入配置模块。
- 验收：模块可在隔离测试环境下独立装配。
2. `DATA_TRANSFORMER_RUNTIME_CONFIG` 工厂改为注入 `ConfigService`。
- 验收：默认值与边界行为不变。
3. 清理直接 `process.env` 读取（仅限本模块改动）。
- 验收：模块内配置读取路径单一。

### 6.4.4 测试补齐
- 新增 `data-transformer.module.spec.ts`：验证显式配置依赖、默认值与边界解析

### 6.4.5 回归验证
- transformer 模块相关单测通过
- 编译通过

## 6.5 M-2：配置读取逻辑重复（P3）

### 6.5.1 背景证据
- `stream-receiver.service.ts:1502`
- `stream-receiver.service.ts:1573`
- `stream-receiver.service.ts:1654`
三处读取同一配置并做相似判定。

### 6.5.2 修复目标
抽取单一配置访问器/判定函数，统一默认值与异常处理，消除行为漂移风险。

### 6.5.3 原子步骤与验收条件
1. 提取私有方法（示例：`isIdentityProvider(provider)`）。
- 验收：三处调用替换完成，重复代码消失。
2. 统一默认值与容错策略。
- 验收：空配置、异常配置下行为一致。
3. 可选一次性缓存（按需，避免过度设计）。
- 验收：无行为变化，代码复杂度可控。

### 6.5.4 测试补齐
- 命中 identity provider/未命中分支双覆盖
- 异常或空配置输入覆盖

### 6.5.5 回归验证
- `stream-receiver.service.spec.ts` 相关测试通过
- stream-receiver 相关单测通过

- 模块配置依赖显式，重复配置读取消除
- 定向测试、相关模块测试、全量单测、编译全部通过
