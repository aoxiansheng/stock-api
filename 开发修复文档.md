### E：影响面分析（确认问题）

1. 队列状态查询路径（Critical）
- 涉及文件：`src/core/03-fetching/stream-data-fetcher/services/stream-recovery-worker.service.ts`
- 依赖链路：gateway `get-recovery-status` -> worker `getClientRecoveryStatus` -> 队列访问
- 潜在风险：高并发轮询时 Redis/Node 负载放大，状态接口可用性下降

2. 恢复窗口语义（Important）
- 涉及文件：`src/core/01-entry/stream-receiver/gateway/stream-receiver.gateway.ts`、`src/core/01-entry/stream-receiver/constants/stream-receiver-timeouts.constants.ts`、`src/core/01-entry/stream-receiver/services/stream-receiver.service.ts`
- 依赖链路：客户端 `request-recovery` -> gateway 预检 -> service 重连策略
- 潜在风险：客户端无法区分“窗口超限”与“启动失败”，重试策略偏差

3. 取消后状态一致性（Important）
- 涉及文件：`src/core/03-fetching/stream-data-fetcher/services/stream-recovery-worker.service.ts`
- 依赖链路：`cancelRecoveryJob` -> 状态查询 API
- 潜在风险：取消后短时间返回旧状态，前端轮询误判

4. 状态容器淘汰策略（Important）
- 涉及文件：`src/core/03-fetching/stream-data-fetcher/services/stream-recovery-worker.service.ts`
- 潜在风险：长周期运行下状态容器增长，内存占用上升

### F：原子级修复计划（函数粒度）

1. 统一恢复窗口语义
- 背景：gateway 与 service 窗口口径不一致
- 目标：统一使用 `RECOVERY_WINDOW_MS`，并明确返回窗口超限语义
- 涉及文件：`stream-receiver.gateway.ts`、`stream-response.utils.ts`、`stream-receiver.gateway.recovery.spec.ts`
- 原子步骤：
1. gateway 使用 `STREAM_RECEIVER_TIMEOUTS.RECOVERY_WINDOW_MS` 预检
2. `recoveryWindowExceeded` 支持返回 `maxWindowMs` 上下文
3. 增加窗口超限单测并断言不触发重连
- 验收条件：窗口超限请求返回 `RECOVERY_WINDOW_EXCEEDED` 且 `handleClientReconnect` 未调用

2. 移除全量扫描快照，改事件驱动内存态
- 背景：状态查询路径存在高频全量扫描
- 目标：`getClientRecoveryStatus` 仅读取内存态，不再触发队列全量扫描
- 涉及文件：`stream-recovery-worker.service.ts`、`stream-recovery-worker.service.status.spec.ts`
- 原子步骤：
1. 删除快照扫描相关字段/方法
2. 在 submit/start/finish/cancel 生命周期维护 per-client 状态
3. 调整单测，断言查询不会触发 `queue.getJobs`
- 验收条件：状态查询行为正确，且无 `queue.getJobs` 调用

3. 消除 cancel 后状态短暂陈旧
- 背景：取消任务后状态可能滞后
- 目标：取消动作立即反映到 `pendingJobs/recoveryActive`
- 涉及文件：`stream-recovery-worker.service.ts`、`stream-recovery-worker.service.status.spec.ts`
- 原子步骤：
1. `cancelRecoveryJob` 成功后调用 `markRecoveryCancelled`
2. `markRecoveryCancelled` 立即更新状态与 `lastRecoveryTime`
3. 增加“取消后立即查询”断言
- 验收条件：取消后第一次查询即返回一致状态

4. 增加状态容器淘汰策略
- 背景：状态 Map 只增不减
- 目标：空闲状态按 TTL 回收
- 涉及文件：`stream-recovery-worker.service.ts`
- 原子步骤：
1. 新增 `clientStatusIdleTtlMs`、`clientStatusCleanupIntervalMs`
2. 增加 `cleanupStaleClientStatuses` 周期性懒清理
3. 在状态读写入口触发清理
- 验收条件：状态管理具备回收机制，不引入额外队列扫描
