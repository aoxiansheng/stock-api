## 原子修复计划

### C1 `unsubscribeStream()` 状态一致性修复

- 背景:
  - 当前实现先删除 `clientStateManager` 中的订阅，再触发上游 release、即时退订和房间退出
  - 任一步骤失败都不会恢复已删状态
- 修复目标:
  - 保持“引用计数计算仍基于已移除状态”的前提下，引入显式补偿
  - 任何失败都能恢复本地订阅并取消待退订任务
- 涉及文件:
  - `src/core/01-entry/stream-receiver/services/stream-receiver.service.ts`
  - `test/unit/core/01-entry/stream-receiver/services/stream-receiver.service.spec.ts`
- 影响面:
  - `unsubscribeStream()`
  - `UpstreamSymbolSubscriptionCoordinatorService.scheduleRelease()`
  - `clientStateManager` 的 upstream 引用计数
  - gateway 订阅失败回滚链路
- 原子开发步骤:
  - 在 `unsubscribeStream()` 中记录是否已执行本地状态移除、即时上游退订列表、计划退订列表
  - 保留“先移除状态再计算 release”的顺序，但把后续外部副作用包裹到可补偿事务内
  - 在 `catch` 中执行补偿:
    - 重新 `addClientSubscription()`
    - 取消待退订 `cancelPendingUnsubscribe()`
    - 对已经即时退订成功的 symbol 执行 `subscribeToSymbols()` 补偿
  - 整理 `scheduleRelease()` 回调缩进与异常传播，使回调可读且不隐藏错误
- 每一步验收条件:
  - 单元测试:
    - 上游即时退订失败后，`addClientSubscription()` 被回滚调用
    - room 退出失败后，`addClientSubscription()` 被回滚调用，待退订被取消
    - 非最后订阅者路径仍不触发上游退订
  - 回归测试:
    - 现有 `stream-receiver.service.spec.ts` 全通过
- 测试补齐策略:
  - 在 `stream-receiver.service.spec.ts` 新增两个失败补偿用例

### I5 gateway owner lease 自动查找回归修复

- 背景:
  - `handleSubscribe()` 目前只处理显式 `sessionId` 路径，缺少无 `sessionId` 的 owner lease 查找
  - 现有 2 个 gateway 单测已失败
- 修复目标:
  - 对“已认证 + 单 symbol + 指定 preferredProvider + 未显式传 `sessionId`”路径恢复 owner lease 自动查找
  - owner lease 命中时自动绑定并使用 lease 的 provider/capability
  - 查找异常时按既有测试要求降级为普通订阅
- 涉及文件:
  - `src/core/01-entry/stream-receiver/gateway/stream-receiver.gateway.ts`
  - `test/unit/core/01-entry/stream-receiver/gateway/stream-receiver.gateway.subscribe.spec.ts`
- 影响面:
  - chart-intraday 运行期租约复用
  - subscribe 入口的前置订阅编排
- 原子开发步骤:
  - 提取 `maybeResolveChartIntradayOwnerLease()` 私有方法或在 `handleSubscribe()` 内加入最小局部逻辑
  - 仅在满足触发条件时调用 `findRealtimeOwnerLease()`
  - 命中租约后重写 `subscribeData.symbols/preferredProvider/wsCapabilityType`
  - 在订阅成功后调用 `bindRealtimeClientToSession()`
  - 查找异常时记录 `warn` 并继续普通订阅
- 每一步验收条件:
  - 单元测试:
    - 现有 2 个失败用例恢复通过
    - 多 symbol 和未认证连接不触发自动绑定
  - 回归测试:
    - `stream-receiver.gateway.subscribe.spec.ts` 全通过
- 测试补齐策略:
  - 复用现有 gateway spec，不新增新测试文件

### I1 `setupDataReceiving()` 幂等化

- 背景:
  - 同一连接被重复订阅或重连时，会重复注册回调
- 修复目标:
  - 同一 `connection` 只绑定一次 ingress 回调
  - 不改变 `handleIncomingData()` 现有输入输出契约
- 涉及文件:
  - `src/core/01-entry/stream-receiver/services/stream-ingress-binding.service.ts`
  - `test/unit/core/01-entry/stream-receiver/services/stream-ingress-binding.service.spec.ts`
- 影响面:
  - `subscribeStream()`
  - `executeClientReconnect()`
  - 批处理入口事件量
- 原子开发步骤:
  - 在 `StreamIngressBindingService` 内引入连接级幂等标记容器
  - `setupDataReceiving()` 检查已绑定连接后直接返回
  - 保持现有 onData/onError/onStatusChange 注册逻辑不变
  - 顺手补强数组 item 的运行时形状过滤，避免 `null/undefined` 抛错吞整批
- 每一步验收条件:
  - 单元测试:
    - 同一连接多次调用 `setupDataReceiving()` 时只注册一次回调
    - 脏数组输入不抛异常且能过滤出合法 symbol
  - 回归测试:
    - `stream-ingress-binding.service.spec.ts` 全通过
- 测试补齐策略:
  - 在既有 spec 中增加幂等测试与脏数组测试

### I2 + M1 symbol/provider 边界规范化修复

- 背景:
  - `mapSymbols()` 异常时回退原始值，破坏标准 symbol 不变量
  - `preferredProvider` 查注册表前未规范化
- 修复目标:
  - 移除不必要的回退逻辑，失败即抛异常
  - provider 查找统一走规范化值
- 涉及文件:
  - `src/core/01-entry/stream-receiver/services/stream-subscription-context.service.ts`
  - `src/core/01-entry/stream-receiver/services/stream-provider-resolution.service.ts`
  - `test/unit/core/01-entry/stream-receiver/services/stream-subscription-context.service.spec.ts`
  - `test/unit/core/01-entry/stream-receiver/services/stream-provider-resolution.service.spec.ts`
- 影响面:
  - subscribe / unsubscribe / reconnect 的 symbol 解析
  - provider registry 查询
- 原子开发步骤:
  - `mapSymbols()` 捕获异常后改为抛出业务异常，不再回退原始值
  - `resolveSymbolMappings()` 对 identity-provider 分支过滤空 canonical symbol，防止空 key 漏出
  - `validatePreferredProviderForStream()` 先规范化 provider 名称，再查 `getProvider/getCapability`
  - 调整对应单测期待
- 每一步验收条件:
  - 单元测试:
    - 映射失败时抛异常，不再返回原始数组
    - identity-provider 分支遇到空白 symbol 时不输出空字符串
    - `preferredProvider = "LongPort"` 仍能正确命中 registry
  - 回归测试:
    - `stream-subscription-context.service.spec.ts`
    - `stream-provider-resolution.service.spec.ts`
- 测试补齐策略:
  - 直接改已有断言与新增 1-2 个边界用例

### I3 重连协调器去兼容回退与收敛 fan-out

- 背景:
  - 当前使用 `as any` 兼容探测，且对批量 client 直接 fire-and-forget
- 修复目标:
  - 直接依赖 `StreamClientStateManager` 明确方法
  - 引入最小成本的分批串行执行，避免无界 fan-out
- 涉及文件:
  - `src/core/01-entry/stream-receiver/services/stream-reconnect-coordinator.service.ts`
  - `test/unit/core/01-entry/stream-receiver/services/stream-reconnect-coordinator.service.spec.ts`
- 影响面:
  - `detectReconnection()`
  - provider 断连补偿
  - heartbeat 超时补偿
- 原子开发步骤:
  - 删除 `as any` 能力探测，直接调用 `getClientsWithHeartbeatTimeout()` / `getClientsByProvider()`
  - 提取 `processClientsInBatches()` 私有函数，按固定 batch 大小分批 `Promise.allSettled`
  - 对单轮 client 列表先去重，避免同轮重复调度
  - 保留现有“最多 50 / 500”截断语义
- 每一步验收条件:
  - 单元测试:
    - `detectReconnection()` 仍能调用 `handleReconnection()`
    - 不再依赖 `as any` 分支
    - 同一 client 重复出现时只调度一次
  - 回归测试:
    - `stream-reconnect-coordinator.service.spec.ts` 全通过
- 测试补齐策略:
  - 在既有 spec 增加去重与分批调度断言

### I4 限流 fail-open 改为 fail-closed

- 背景:
  - 限流依赖异常时直接允许通过
- 修复目标:
  - 连接限流依赖异常时拒绝订阅请求，并返回明确业务异常
- 涉及文件:
  - `src/core/01-entry/stream-receiver/services/stream-receiver.service.ts`
  - `test/unit/core/01-entry/stream-receiver/services/stream-receiver.service.spec.ts`
- 影响面:
  - subscribe 入口门禁
  - API Key / IP 级连接保护
- 原子开发步骤:
  - `checkConnectionRateLimit()` 改为在依赖异常时抛业务异常，或返回明确拒绝态
  - `subscribeStream()` 统一处理“超限”和“限流依赖不可用”两种拒绝路径
  - 保持 `RateLimitService` 未注入时的显式日志，但不再对“已注入却异常”放行
- 每一步验收条件:
  - 单元测试:
    - `checkRateLimit` 抛错时，`subscribeStream()` 被拒绝
    - 超限时仍短路且不触发 `validateSubscribeRequest`
  - 回归测试:
    - `stream-receiver.service.spec.ts` 相关用例通过
- 测试补齐策略:
  - 在既有 spec 中新增“rate limit service throw”用例
